## 1. 文档说明

本文档用于承接现有需求规格说明书与技术栈选型说明书，给出 EduWeave 项目的后端概要设计方案，明确后端系统职责边界、总体架构、业务模块划分、核心任务编排、数据与版本策略、外部集成方式以及接口分组原则，为后续数据库详细设计、接口文档设计、后端脚手架搭建和任务实现提供统一依据。

本文档定位于“后端概要设计”层，重点解决“系统应该如何拆分和组织”的问题，不直接替代以下文档：

- 数据库表结构设计文档
- API 详细设计文档
- Prompt 组装与结构化输出约束文档
- 部署与运维文档

## 2. 设计依据与约束

### 2.1 设计依据

本方案直接基于以下现有文档与样例材料进行设计，不新增脱离上下文的业务设定：

- `docs/教育赛题要求.md`
- `docs/教育赛题需求规格说明书.md`
- `docs/技术栈选型说明书.md`
- `教育赛题/教材文档/` 下教材样本
- `教育赛题/学情分析/` 下学情样本
- `教育赛题/示例文件/` 下教案与教学大纲参考样本

### 2.2 已知约束

结合现有文档，本期后端设计需遵循以下约束：

- 系统核心主链路必须围绕“上传 -> 解析 -> 结构化 -> 规划 -> 生成 -> 校验 -> 交付”展开。
- 核心业务数据以 `MySQL 8.0` 为中心，文件资产统一存储于 `华为云 OBS`。
- 教材解析入口采用 `MinerU`，课程规划、教案、课件、测练与覆盖率报告依赖大模型与结构化生成能力。
- 后端主栈采用 `Python 3.12 + FastAPI + Pydantic v2 + SQLAlchemy 2 + Alembic`。
- 长耗时链路必须采用 `Celery + Redis` 异步化处理，前端在 P0 阶段通过轮询获取任务状态。
- 用户角色本期以教师为主，不引入复杂组织、多租户和审批流设计。
- P0 阶段必须提供基础用户认证能力，至少覆盖用户名密码登录、获取当前用户信息和受保护接口鉴权。
- 本期默认优先满足比赛演示与最小可用交付，并将 Milvus 检索能力纳入主链路实现；SSE、多角色协同等增强项不作为主链路阻塞点。

## 3. 设计目标与原则

### 3.1 设计目标

后端概要设计的目标不是单纯提供一组接口，而是构建一套可持续演进的教学资源生产后端骨架，使其具备以下能力：

- 稳定承接教材、学情、解析结果、知识结构、教学规划与成果物的完整生命周期。
- 让上传、解析、结构化、生成和校验各阶段的数据可以相互关联、相互追溯。
- 让长流程任务具备异步执行、失败重试、局部修正和版本留痕能力。
- 为后续数据库设计、接口设计和模块编码提供清晰边界，减少返工。

### 3.2 设计原则

- 主链路优先：优先支持赛题要求中的核心闭环，不为边缘能力引入过重架构。
- 结构化优先：大模型与解析输出必须先落为结构化中间层，再驱动课件、测练和报告生成。
- 版本优先：教材、学情、解析结果、知识结构和生成产物均需要有明确版本基线。
- 异步优先：教材解析、学情抽取、知识抽取、课程规划、课件生成、测练生成、覆盖率分析均按任务化设计。
- 领域优先：后端目录和服务按业务域划分，而不是按技术分层平铺。
- 可追溯优先：任一成果物都应能回溯到输入文件、知识点、学情依据、模型参数和任务链路。
- P0 与 P1 分离：P0 保证主链路跑通，P1 再扩展语义检索、实时推送、复杂模板中心等增强能力。

## 4. 后端职责边界

### 4.1 后端负责内容

本期后端负责以下能力建设：

- 项目基础信息管理与项目级资源归档。
- 教材文件、学情文件与参考文件的上传校验、存储编排和元数据管理。
- 教材解析任务、学情抽取任务、知识抽取任务、课程规划任务、资源生成任务、覆盖率分析任务的提交、执行、追踪与重试。
- 教材章节树、知识点矩阵、教学目标摘要、学情画像、课程大纲、教案、测评蓝图、课件结构、题目与覆盖率报告的结构化存储。
- 成果物导出文件与结构化中间结果的统一归档。
- 任务日志、操作审计、来源追溯与版本链路管理。
- 基础认证与角色鉴权能力。

### 4.2 后端不直接负责内容

以下内容不作为本期后端设计的直接实现目标，但需预留扩展空间：

- 多租户组织管理、审批流与企业级权限体系。
- 学生端在线答题、自动判分与学习闭环。
- 复杂实时协同编辑。
- 高度自由的课件视觉编排引擎。
- 对 MinerU、模型厂商和向量库内部算法的深度定制。

## 5. 总体架构设计

### 5.1 运行架构

后端整体采用“API 服务 + 异步 Worker + 关系型数据库 + 对象存储 + 外部基础设施组件”的组合模式。推荐运行关系如下：

```text
React 前端工作台
    |
    v
FastAPI API 服务
    |-- 读写 MySQL
    |-- 读写 OBS
    |-- 提交 Celery 任务到 Redis
    |
    v
Celery Worker
    |-- 调用 MinerU 解析教材
    |-- 调用大模型服务完成知识抽取与生成
    |-- 生成或归档 PPTX/DOCX 等成果物
    |-- 回写 MySQL 与 OBS
    |
    +-- 调用 Milvus 执行向量写入与检索
```

该架构满足以下目标：

- API 进程专注于请求校验、状态查询、任务触发与结构化读写，避免阻塞。
- Worker 进程负责高耗时任务，便于失败重试、资源隔离和阶段拆分。
- MySQL 作为唯一业务事实源，OBS 作为文件资产中心，避免状态分散。
- 外部基础设施组件通过适配层接入，降低直接耦合度。

### 5.2 逻辑分层

后端逻辑上划分为五层：

| 层级 | 组成 | 核心职责 |
| --- | --- | --- |
| 接入层 | Router、依赖注入、中间件 | 提供 `/api/v1` REST 接口、鉴权、参数校验、响应封装 |
| 应用层 | Service、任务编排器 | 组织业务流程，控制版本生效、任务提交、状态流转 |
| 领域持久层 | Repository、SQLAlchemy Model | 负责实体持久化、查询聚合、分页筛选和事务边界 |
| 基础设施层 | OBS 适配、MinerU 适配、LLM 适配、Redis/Celery、日志组件、Milvus 适配 | 屏蔽外部基础设施差异，对外提供统一能力 |
| 异步执行层 | Worker、任务处理器、重试策略 | 执行解析、抽取、生成、校验等长链路任务 |

### 5.3 目录组织原则

后端目录承接现有技术栈文档中的推荐结构，采用“按业务模块划分，模块内局部分层”的方式组织：

- `backend/app/core`：放置配置、数据库、异常、日志、安全、中间件等核心支撑能力。
- `backend/app/shared`：放置跨业务域复用的基础设施能力，如 `storage`、`mineru`、`llm`、`queue`、`vector`、`utils`。
- `backend/app/modules`：按业务域放置 `router.py`、`service.py`、`repository.py`、`schemas.py`、`models.py` 与模块内测试。
- `backend/app/worker.py`：Worker 启动入口。
- `backend/migrations`：Alembic 迁移。
- `backend/tests`：跨模块集成测试。
- `backend/docs`：后端内部实现说明、接口约束、Prompt 输出规范等后续文档。

## 7. 核心流程与任务编排设计

### 7.1 项目创建与文件入库流程

项目是本期后端的业务上下文容器和工作台入口。教材、学情、解析结果、规划结果、生成产物和任务记录均在项目下做归属管理，但对应领域逻辑仍由各业务模块分别负责。项目创建后，文件入库流程建议如下：

1. 用户通过 `auth` 模块完成登录认证，获取访问令牌。
2. 前端提交项目基础信息，后端创建项目记录。
3. 前端上传教材或学情文件，后端完成格式、大小、哈希和基础完整性校验。
4. 后端将原始文件写入 OBS，并在 MySQL 中记录统一文件对象元数据。
5. 若为教材文件，则创建教材版本记录并挂接到项目。
6. 若为学情文件，则创建学情文件记录，并触发后续学情抽取任务。

P0 阶段建议采用“后端接收文件后再转存 OBS”的模式，理由如下：

- 实现简单，便于快速联调。
- 上传成功后可以立即完成哈希计算、元数据登记与业务归档。
- 便于统一做文件校验和错误处理。

后续若大文件吞吐压力增大，可在 P1 阶段演进为前端直传 OBS + 回调登记模式。

### 7.2 教材解析流程

教材解析采用“全量解析为主，局部重解析为辅”的双层设计：

1. 用户在教材版本上发起解析。
2. API 服务创建解析任务记录，并把任务投递到 `parsing` 队列。
3. Worker 调用 `shared.mineru` 适配层，输出标准化的解析包。
4. 解析包中的结构化内容、中间文件和页图切片写入 OBS。
5. MySQL 中写入新的解析版本、页级结构、块级结构、异常检测结果。
6. 系统把该解析版本标记为“待复核”或“可用”状态，供知识抽取使用。

针对复杂教材，需要额外支持以下机制：

- 异常检测：识别缺页、乱码、块丢失、公式截断、表格断裂等问题。
- 页级重解析：针对异常页触发局部任务，而不是重跑整本教材。
- 人工复核：对公式、跨页表格、实验图表等复杂内容提供异常标记、对照预览和复核入口。
- 人工修正：用户修正后的结果不直接覆盖原始记录，而是生成新的解析版本或解析补丁。

### 7.3 学情抽取流程

学情抽取与教材解析分离，避免两类输入耦合。流程建议如下：

1. 学情文件上传完成后创建学情抽取任务。
2. Worker 从 doc/docx 中抽取文本，按样例文档结构解析基础字段和自由文本描述。
3. 将多学科内容拆分为多条“学生-学科学情画像”记录。
4. 保存结构化字段、摘要文本、抽取置信信息和原文证据片段。
5. 若用户进行人工修正，则形成新的学情版本并更新项目下的有效画像。

为适配现有样本，P0 阶段建议把学情数据拆为两层：

- 学情文件层：记录原始文档与整体抽取状态。
- 学情画像层：以“学生标识 + 学科”为粒度存储能力标签、成绩、行为特点和时间规划。

### 7.4 知识结构化流程

知识结构化依赖“已确认可用”的解析版本。流程建议如下：

1. 选择教材解析版本作为输入基线。
2. 基于解析 Markdown 或页级内容整理教材 `semantic_chunk`，保留来源页码、解析块、坐标和资源引用。
3. 从语义块中提取章节树、课文、知识点、重难点、教学目标摘要与来源证据。
4. 建立知识点与章节、语义块、页码、块坐标、原文片段的映射关系。
5. 生成知识版本，供课程大纲、教案、测评蓝图与课件生成复用。

为兼顾查询效率与演示可用性，P0 阶段建议：

- 对章节树、知识点建立可筛选的关系型字段。
- 对教材语义块建立 `semantic_chunk`，后续教材片段向量化优先使用语义块文本；MinerU `parse_block` 继续用于版面追溯。
- 教学目标摘要、重难点、推荐教学活动等内容先放入 `knowledge_version.summary_json` 或下游 `content_json`。
- 对复杂证据、原文片段、推荐难度、标签集合采用 JSON 字段保存。
- 知识点的人工增删改不直接改写旧版本，而是生成新的知识版本。

### 7.5 课程规划与资源生成流程

课程规划与资源生成整体上采用“先规划、后生成、再校验”的顺序编排：

1. 选定项目、教材版本、知识版本、学情版本和目标章节范围。
2. 生成课程大纲版本。
3. 基于课程大纲的 `lesson_sessions` 拆分生成多份课次教案，先写入课次级中间结果。
4. 基于 `generation_batch` 汇总课程大纲和本批次全部教案，执行自动覆盖率分析。
5. 用户按需基于课程大纲创建测评任务，生成测评蓝图、试卷和题目。
6. 用户按需基于指定教案创建课件任务，生成课件结构并归档 PPTX。

这里不建议直接把全部环节揉成一个黑盒任务，而是采用“`pipeline` + 批次 + 子任务”的方式：

- `pipeline` 负责创建和维护 `generation_batch`。
- 批次负责记录本次生成的统一输入基线。
- 自动子任务负责课程大纲、教案和覆盖率；测评与课件由用户主动创建按需任务。
- 教案生成使用课次中间结果表做断点续跑，全部课次成功后再形成正式教案，避免前端看到失败批次中的部分正式教案。
- 任一子任务失败时，可单独重试；多课时教案失败任务可通过任务中心手动重试入口只补失败或缺失课次。

### 7.6 任务状态与队列设计

为与现有需求文档保持一致，任务中心对外统一暴露以下状态：

- `pending`：待处理
- `processing`：处理中
- `success`：成功
- `partial_success`：部分成功
- `failure`：失败
- `cancelled`：已取消

每个任务还应补充以下字段，便于前端工作台展示：

- `task_type`：任务类型，如教材解析、学情抽取、知识抽取、课件生成等
- `current_stage`：当前执行阶段
- `progress_percent`：进度百分比
- `retry_count`：重试次数
- `last_error_code`
- `last_error_message`
- `started_at`
- `finished_at`

队列划分建议如下：

- `parsing_queue`：教材解析、局部重解析、异常检测
- `profile_queue`：学情抽取、学情重算
- `knowledge_queue`：章节与知识点抽取
- `generation_queue`：课程大纲、教案、测评、课件、覆盖率分析等生成与质控任务
- `analysis_queue`：未来可拆分队列，当前代码尚未单独启用

P0 阶段即使先用单 Worker，也建议保留多队列划分，便于后续按资源消耗做隔离扩容。

## 8. 数据与版本设计

### 8.1 数据存储策略

后端数据建议采用“关系型主干字段 + JSON 结构化内容 + OBS 文件对象”的组合策略：

- 关系型字段负责对象主键、归属关系、状态、版本号、筛选条件、时间戳和索引字段。
- JSON 字段负责保存章节树、课程大纲结构、教案流程、幻灯片结构、题目结构和覆盖率明细等层级内容。
- OBS 负责保存原始教材、学情文档、MinerU 中间产物、页图切片、PPT、DOCX、PDF 和快照文件。

该策略的优点是：

- 既能满足列表、筛选、统计、分页等常规后端查询需求。
- 又能避免在 P0 阶段把复杂层级对象过度范式化，降低表设计复杂度。
- 仍然保留后续针对高频对象进一步拆表优化的空间。

### 8.2 核心实体建议

本期数据库详细设计建议至少覆盖以下实体：

| 实体分类 | 建议实体 | 设计说明 |
| --- | --- | --- |
| 用户与项目 | `sys_user`、`project` | `sys_user` 承载教师基础身份信息；`project` 作为业务上下文容器 |
| 文件与版本 | `file_object`、`textbook_version`、`learner_profile_file`、`learner_profile_version` | `file_object` 统一承载 OBS 元数据；P0 阶段教材先按项目内版本建模，不额外拆 `textbook_document` |
| 学情画像 | `learner_profile_record` | 以“学生标识 + 学科”为粒度维护结构化画像 |
| 任务中心 | `task_record`、`task_step_record` | 统一承载解析、抽取、生成、分析任务；写入协议由 `shared.queue` 统一约束 |
| 教材解析 | `parse_version`、`parse_page`、`parse_block`、`parse_issue` | 支持全量解析、局部重解析、异常记录与人工修正版本链 |
| 知识结构 | `knowledge_version`、`chapter_node`、`semantic_chunk`、`knowledge_point`、`knowledge_evidence` | 从生效解析版本抽取结构化知识资产 |
| 课程规划 | `curriculum_plan`、`lesson_plan`、`assessment_blueprint` | 以版本化方式保存课程大纲、教案、测评蓝图 |
| 成果物 | `generation_batch`、`courseware_result`、`paper_result`、`question_item`、`coverage_report` | 以批次维度绑定同一次生成的输入基线与输出结果 |
| 追溯审计 | `generation_trace`、`audit_log` | 分别承载内容来源链和操作行为留痕 |

### 8.3 版本基线设计

版本一致性是本项目后端设计的核心。建议采用以下规则：

#### 8.3.1 文件版本

- 同一教材允许多次上传，形成多个 `textbook_version`。
- 同一学情文件允许修订，形成多个 `learner_profile_version`。
- 原始文件不可被覆盖，只能新增版本。
- P0 阶段教材版本直接挂接项目，不额外拆分教材主档。

#### 8.3.2 解析版本

- 每次全量解析产生一个新的 `parse_version`。
- 每次页级重解析也产生新的 `parse_version`，并记录 `parent_parse_version_id`。
- 人工修正不直接覆盖旧解析结果，而是生成新的解析版本或解析修订记录。
- 仅“已确认可用”的解析版本才能进入知识抽取流程。

#### 8.3.3 知识版本

- 每个 `knowledge_version` 必须绑定唯一的 `parse_version`。
- 知识点人工修正、归并或删除后，生成新的知识版本。

#### 8.3.4 规划与生成版本

- `curriculum_plan`、`lesson_plan`、`assessment_blueprint` 均按版本保存。
- `generation_batch` 记录一次生成批次所使用的知识版本、学情版本、章节范围、模板快照、测评策略、编排选项，以及生成得到的课程大纲和首份教案引用；批次下全部教案、测评结果和课件结果通过各自表按批次反查。
- 教材版本与解析版本通过 `knowledge_version -> parse_version -> textbook_version` 关系回溯，不在批次表中重复存储。
- 模型参数、Prompt 版本和任务级输入证据摘要统一写入批次快照 JSON 与任务记录，不再为每类参数单独拆字段。
- `courseware_result`、`paper_result`、`coverage_report` 均挂接在 `generation_batch` 下，确保同一批次成果物使用统一基线。
- 只有 `pipeline` 模块允许同时接收跨阶段版本作为批次输入；下游模块仅显式接收直接父版本或 `generation_batch_id`。

上述设计可以解决需求规格中提出的“一次生成任务的多个成果物应使用统一目标范围与版本基线”的一致性要求。

### 8.4 人工修正生效策略

为避免直接覆盖历史结果，建议采用“新增版本生效”策略：

- 原始解析结果永远保留，便于回溯和差异比对。
- 人工修正后的结果写入新版本，不直接覆盖旧版本。
- P0 阶段不引入统一生效指针；教材和学情默认引用由 `project` 维护，解析与知识结果由下游对象显式绑定或按最近确认版本回显。
- 若需要回退，重新选择历史版本或重建下游绑定关系即可，不必恢复旧数据。

## 9. 外部依赖与集成设计

### 9.1 MinerU 集成设计

`shared.mineru` 适配层负责屏蔽 MinerU 的调用细节，对业务层暴露统一输入输出：

- 输入：教材文件对象、解析策略、页范围。
- 输出：标准化解析包，至少包含页面文本、标题层级、结构块、图片/表格/公式资源索引、异常信息和原始中间文件引用。

适配层必须承担以下职责：

- 统一不同解析模式下的返回结构。
- 为异常页重解析提供输入组装能力。
- 记录调用耗时、策略版本、失败原因和原始返回摘要。
- 将复杂错误映射为后端统一业务错误码。

### 9.2 OBS 集成设计

`shared.storage` 负责统一 OBS 读写，并对业务层隐藏 Bucket、Endpoint、AK/SK 等细节。对象路径建议按项目、对象类型和版本规划，例如：

| 对象类型 | 建议对象路径 |
| --- | --- |
| 原始教材 | `projects/{project_id}/textbooks/{textbook_version_id}/source.pdf` |
| 原始学情 | `projects/{project_id}/learner-profiles/{profile_file_id}/versions/{version_no}/source.docx` |
| 解析中间产物 | `projects/{project_id}/textbooks/{textbook_version_id}/parse/{parse_version_no}/mineru/...` |
| 原页图与切片 | `projects/{project_id}/textbooks/{textbook_version_id}/parse/{parse_version_no}/assets/...` |
| 课件成果物 | `projects/{project_id}/generation/{generation_batch_id}/courseware/...` |
| 测练成果物 | `projects/{project_id}/generation/{generation_batch_id}/assessment/...` |
| 覆盖率报告 | `projects/{project_id}/generation/{generation_batch_id}/coverage/...` |

统一对象路径规划的价值在于：

- 项目级归档和清理更简单。
- 同一业务对象的多版本文件可以自然聚合。
- 任务失败时便于定位中间产物和结果文件。

### 9.3 大模型服务集成设计

`shared.llm` 适配层应提供统一的任务调用入口，不让业务模块直接拼接供应商 SDK。建议设计要点如下：

- Prompt 当前由知识抽取、课程大纲、教案、测评、课件等任务在代码内按上下文组装；后续如模板数量继续扩张，再迁移到文件化模板管理。
- 每个任务约束固定结构化输出 Schema，输出经 Pydantic 校验后方可入库。
- 每次任务记录输入基线、结构化输出、模型调用失败原因和必要的证据摘要；Prompt 版本字段当前以批次快照或任务元数据预留为主。
- 对超时、结构化输出不完整、字段缺失等场景支持重试或回退。

### 9.4 向量检索集成设计

这里的“向量检索”不是指调用某个第三方托管检索 API，而是指后端对接独立部署的向量数据库组件。本项目接入 Milvus，应理解为：

- Milvus 是系统外部运行的基础设施组件，职责是存储向量、建立索引并执行检索。
- 后端代码中实现的是 `shared.vector` 适配层，负责封装 `pymilvus` 调用、统一接口、处理元数据映射与错误转换。
- 教材片段向量的 MySQL 主映射对象为 `semantic_chunk`，云端集合名为 `semantic_chunk_vector`。
- 业务模块不直接依赖具体 Milvus SDK，而是通过适配层调用向量写入、删除、检索和健康检查能力。
- 若部署形态为 Zilliz 提供的 Milvus 集群或托管环境，但实际接入协议仍是 Milvus 原生能力，则架构上仍按“向量数据库基础设施依赖”处理，而不是按“托管 API 服务”处理。
- 当向量库临时不可用时，知识召回流程应进入告警、重试或受控降级机制。

因此，在架构表达上更准确的说法是：

- 向量检索能力的核心实现由 Milvus 这类向量数据库组件承担。
- 后端代码负责实现对该组件的接入适配、调用编排与业务封装。
- 本项目代码不会在应用层自行实现一个向量数据库内核，只会实现与 Milvus 的集成逻辑。

### 9.5 成果物生成设计

课件、教案、作业和试卷的最终导出仍由后端 Worker 完成，但必须坚持“结构化中间层优先”原则：

- 课程大纲、教案、幻灯片结构、题目结构先落 MySQL。
- PPTX 当前通过 Raccoon PPT OpenAPI 生成并下载归档，DOCX 当前通过 `python-docx` 直接渲染。
- PDF 导出暂未实现，PDF 相关能力主要用于教材解析、页图预览和局部重解析。
- 导出文件写入 OBS，数据库仅存文件引用、摘要信息和版本信息。

## 10. API 概要设计

### 10.1 接口设计原则

后端接口设计建议遵循以下原则，并与现有 FastAPI 规范保持一致：

- 统一使用 `/api/v1/**` 前缀。
- 路由路径使用资源风格，路径参数采用 `snake_case`。
- 统一使用 JWT Bearer 认证，公开接口仅保留登录等必要入口。
- 统一返回响应壳结构，包含 `success`、`code`、`message`、`data`、`timestamp`、`request_id`。
- 列表接口统一采用分页结构 `items + pagination`。
- 长耗时任务接口只负责“创建任务”，不在请求内阻塞等待最终结果。
- 每个接口都需要补充 `summary`、`description`、`operation_id` 等文档注解。

### 10.2 路由分组建议

| 路由分组 | 说明 |
| --- | --- |
| `/api/v1/auth` | 登录、当前用户信息等基础认证接口 |
| `/api/v1/projects` | 项目创建、列表、详情和基础信息维护 |
| `/api/v1/projects/{project_id}/dashboard` | 项目工作台聚合数据 |
| `/api/v1/projects/{project_id}/active-refs` | 当前教材版本、学情版本默认引用切换 |
| `/api/v1/projects/{project_id}/textbooks` | 项目下教材上传、教材版本列表、版本切换 |
| `/api/v1/projects/{project_id}/learner-profiles` | 学情文件上传、学情版本、画像记录 |
| `/api/v1/textbook-versions/{textbook_version_id}/parse-tasks` | 基于教材版本发起解析、重解析、解析任务列表 |
| `/api/v1/textbook-versions/{textbook_version_id}/parse-versions` | 解析版本详情、页块预览、异常列表 |
| `/api/v1/parse-versions/{parse_version_id}/knowledge-versions` | 基于解析版本查看知识版本、章节树、知识点矩阵 |
| `/api/v1/curriculum-plans` | 课程大纲生成、列表、详情、编辑生效 |
| `/api/v1/lesson-plans` | 教案生成、列表、详情、导出 |
| `/api/v1/assessment-blueprints` | 测评蓝图生成、编辑、详情 |
| `/api/v1/generation-batches` | 批次创建、详情、整批重试、结果摘要 |
| `/api/v1/coursewares` | 课件生成、详情、导出 |
| `/api/v1/papers` | 作业/试卷生成、题目详情、导出 |
| `/api/v1/coverage-reports` | 覆盖率报告查看、告警详情 |
| `/api/v1/tasks` | 统一任务列表、任务详情、步骤进度、错误摘要和失败教案任务重试 |
| `/api/v1/audits` | 后续预留审计日志与追溯查询能力 |

### 10.3 P0 阶段重点接口能力

P0 阶段优先保证以下接口集可用：

- 登录、获取当前用户信息。
- 项目创建、项目详情、项目工作台概览、默认教材/学情版本切换。
- 教材上传、教材版本查看、学情上传、学情画像查看。
- 教材解析任务创建、任务轮询、解析结果预览。
- 知识点矩阵查看与基础人工修正。
- 课程大纲生成、教案生成、批次创建、课件生成、作业或试卷生成。
- 覆盖率报告查看与成果物下载。
- 统一任务中心查询、步骤进度查看和轻量生成追溯。

这组接口足以支撑需求文档中定义的最小可用闭环，也最适合优先进入详细设计。

## 11. 横切能力设计

### 11.1 认证与权限

结合当前赛题范围，P0 阶段采用单教师角色的简化权限策略：

- 用户表维护教师账号与账号状态。
- 接口层通过 JWT + 依赖注入获取当前用户。
- 教师接口负责项目、文件、解析、生成、基础配置和下载等业务操作。
- 多租户、细粒度 RBAC、组织隔离能力放入 P1 以后扩展。

### 11.2 错误码与异常处理

为便于后续实现，建议业务错误码按模块分段设计：

- `1xxx`：认证与权限
- `2xxx`：项目与文件
- `3xxx`：教材与学情
- `4xxx`：解析与人工修正
- `5xxx`：知识结构与课程规划
- `6xxx`：资源生成与覆盖率分析
- `7xxx`：外部服务调用
- `8xxx`：系统级异常

服务层返回业务语义错误，路由层或全局异常处理器再映射到 HTTP 状态码，从而保持业务实现与接口协议解耦。

### 11.3 日志与链路追踪

后端应在 API 与 Worker 两侧同时建立链路追踪能力：

- API 请求生成或透传 `request_id`。
- 任务执行生成 `task_id`，并在后续子任务和外部调用日志中透传。
- 关键日志统一记录 `request_id`、`task_id`、`project_id`、`operator_id`、`module`、`action`。
- 任务失败时，必须保留阶段名称、错误码、错误摘要和必要上下文。

### 11.4 幂等与重试

本项目大量任务具备重复触发风险，因此建议明确以下规则：

- 文件上传以哈希辅助去重，但是否复用旧文件由业务层控制。
- 解析任务对同一教材版本的同一解析策略可以做重复提交保护。
- 生成任务若输入基线、模板和参数完全一致，可提示复用最近成功批次；若用户显式要求重生成，则创建新批次。
- Worker 侧所有重试任务都应只传业务对象 ID，避免在消息体中携带大对象。

### 11.5 一致性保障

为满足需求文档中的一致性要求，建议采用以下控制措施：

- 知识点、课程大纲、教案、课件、试卷、覆盖率报告均必须显式记录来源版本。
- 同一 `generation_batch` 下的全部成果物使用同一组版本引用。
- 覆盖率报告以同一 `generation_batch` 为根，初始分析课程大纲与教案；测评题目和课件页面生成成功后自动刷新，并可由用户手动重新分析。
- 当上游有效版本变更时，系统标记下游成果为“基线过期”，提醒用户重新生成。

## 12. P0 与 P1 边界建议

| 能力 | P0 方案 | P1 扩展方向 |
| --- | --- | --- |
| 文件上传 | 后端接收后转存 OBS | 前端直传 OBS + 回调登记 |
| 任务通知 | 前端轮询任务状态 | SSE 或 WebSocket 推送 |
| 向量检索 | 正式接入 Milvus，提供写入、删除、检索与健康检查 | 混合检索、索引优化、Zilliz 托管形态 |
| 权限体系 | 单教师角色 + 账号状态控制 | 细粒度 RBAC、组织与租户 |
| 模板能力 | 固定模板参数输入 | 模板中心、模板版本管理 |
| 人工复核 | 基础页级纠错与生效 | 协同复核、差异可视化增强 |
| 结果生成 | 单次批量生成 + 单项重试 | 更细粒度增量重生成 |
