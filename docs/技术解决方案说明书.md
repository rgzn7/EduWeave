# EduWeave 技术解决方案说明书

## 1. 文档说明

本文档是一份独立、自包含的 EduWeave 技术解决方案说明，面向赛题交付与方案评审。它从整体设计、关键技术实现路径、系统性能与工程化、应用实现能力、创新与突破点、适用场景与价值六个维度，系统化呈现 EduWeave 如何把「教材 PDF + 学情分析文件」这类死数据，转化为可直接进课堂的智能教学资源。

EduWeave 面向赛题「基于 MinerU 的教材到课堂全链路 AI 教学资源重构」，目标是构建从文档解析、知识结构化、教学内容重组到资源自动生成的全流程生产链路，按教学目标与学情特征自动输出课程大纲、课堂课件、课后作业及测试试卷等标准化教学成果，形成可复用、可扩展的智能教学资源生产体系。

面向读者为方案评审者、技术决策者、部署运维方与后续承建方。阅读本文档可在不深入源码的前提下，建立对系统整体方案、关键技术与工程能力的完整认知。全文章节安排如下：第 2 章整体设计方案，第 3 章关键技术实现路径，第 4 章系统性能与工程化，第 5 章应用实现能力，第 6 章创新与突破点及验证，第 7 章适用场景与价值，第 8 章总结与后续演进。

## 2. 整体设计方案

### 2.1 业务全景与生产闭环

EduWeave 围绕「教材到课堂」的真实教学办公场景，构建一条可复用、可扩展、可追溯的智能教学资源生产链路。系统把传统上耗时、割裂、依赖人工的资源生产过程，收敛为一条贯通的七步闭环：

- 上传：教师上传 PDF 教材与班级学情分析文件（每个学生一份 DOCX）。
- 解析：基于 MinerU 对复杂版式教材做高保真结构化拆解，复原章节、公式、表格、图表。
- 结构化：把解析得到的碎片化版面，重组为「章节树 + 知识点 + 证据映射」的知识图谱，并完成向量化。
- 规划：结合知识图谱与班级学情，生成课程大纲与分课次教案，并建立测评蓝图。
- 生成：自动产出课堂课件（PPT）、课后作业、单元/期末测试试卷等成品资源。
- 校验：生成知识点覆盖率分析报告，可视化考点覆盖与难度分布，实现质量把控。
- 交付：成果物统一管理、预览、Word 导出与签名下载。

闭环的每一步都以「版本」为单位沉淀，前一步的可用版本是后一步的输入基线，从而保证全链路的可追溯与可重跑。

### 2.2 五层逻辑架构

后端遵循「版本优先、异步优先、结构化优先」的设计原则，划分为五个逻辑层，业务模块按域聚合、共用能力下沉。

| 层次 | 职责 | 主要落点 |
| --- | --- | --- |
| 接入层 | HTTP 路由、鉴权、参数校验、统一响应与错误码 | `app/main.py`、各模块 `router.py` |
| 应用层 | 业务编排、用例服务、跨模块协作 | 各模块 `service.py` |
| 领域持久层 | 实体模型、仓储读写、版本与血缘管理 | 各模块 `models.py` / `repository.py` |
| 基础设施层 | 外部服务适配、存储、向量库、配置、日志 | `app/core/**`、`app/shared/**` |
| 异步执行层 | 长流程任务、队列消费、心跳与恢复 | `app/worker.py`、各模块 `tasks.py` |

### 2.3 运行时拓扑

系统运行需要三类应用进程与四类中间件/外部服务协同：

- 应用进程：API 服务（处理 HTTP 请求）、Celery worker（执行解析、抽取、生成、分析等异步任务）、Celery beat（调度僵尸任务回收与课件远程状态复查等周期任务）。
- 数据与中间件：MySQL 8.0（业务数据中心）、Redis（Celery broker/backend）、Milvus/Zilliz（向量检索中心）、华为云 OBS（文件资产中心）。
- 外部 AI 服务：MinerU（教材高保真解析）、OpenAI 兼容 LLM（结构化生成）、独立 Embedding 服务（向量化）、Raccoon PPT OpenAPI（课件排版生成）。

异步任务按业务划分为四个队列：`profile_queue`（学情抽取）、`parsing_queue`（教材解析）、`knowledge_queue`（知识抽取）、`generation_queue`（大纲/教案/课件/试卷/覆盖率生成）；beat 调度的周期任务投递到默认 `celery` 队列。

### 2.4 三阶段数据流与版本血缘

系统按三个阶段推进，阶段间通过数据库实体的版本链严格串联血缘，并由 `generation_batch` 在生成阶段冻结输入基线：

```
教材版本 textbook_version
   └─ 解析版本 parse_version（父链支持局部重解析）
        └─ 知识版本 knowledge_version（章节树 + 知识点 + 语义块 + 向量）
             └─ 生成批次 generation_batch（冻结 knowledge_version + learner_profile_version + 章节范围 + 课次/测评策略）
                  ├─ 课程大纲 curriculum_plan
                  │    └─ 教案 lesson_plan（多课次，含课次级中间结果）
                  │         ├─ 课件 courseware_result（Raccoon PPT）
                  │         └─ 课后作业 homework_blueprint → homework_result → homework_question（按课次）
                  ├─ 测评蓝图 assessment_blueprint → 试卷 paper_result → 题目 question_item
                  └─ 覆盖率报告 coverage_report（汇总以上成果物）
学情版本 learner_profile_version（班级多学生聚合）→ 被 generation_batch 冻结
```

凡涉及内容沉淀的实体均版本化，人工修正生成新版本而非覆盖旧版本，旧版本归档保留。底层数据模型为 MySQL「关系型主干字段 + JSON 层级内容 + OBS 文件对象」的组合策略，约 33 张表，配合 Milvus 向量集合承载语义检索。

### 2.5 模块全景

后端共 18 个业务模块，按阶段与职责归类如下：

| 分类 | 模块 | 核心职责 |
| --- | --- | --- |
| 基础支撑 | `auth` / `project` / `system` / `file_asset` / `task_center` | 认证、项目工作台、系统配置、文件资产下载、任务执行与恢复 |
| 阶段一 输入链 | `textbook` / `parsing` / `learner_profile` | 教材多版本管理、MinerU 解析与复核、班级学情上传与聚合画像 |
| 阶段二 结构化 | `knowledge` | 知识抽取、章节树与知识点、语义块向量化与人工修正 |
| 阶段三 生成链 | `curriculum` / `lesson_plan` / `courseware` / `assessment` / `homework` / `coverage` | 课程大纲、教案、课件、测试试卷、课后作业、覆盖率分析 |
| 编排与展示 | `orchestrator` / `pipeline` / `generation_process` | 一键生成编排、生成批次管理、面向前端的生成过程聚合 |

模块间遵循单向依赖与「跨模块经由对方 service 协作、外部依赖统一走 shared」的协作规则，并明确同步（项目、文件、详情、下载地址）与异步（解析、抽取、生成、分析）的边界。

## 3. 关键技术的实现路径

本章按链路顺序给出后端关键技术的实现路径与代码落点（路径相对 `backend/`）。

### 3.1 非结构化教育知识的高保真提取（MinerU 接入）

教材解析以 MinerU 真实接入为核心，落点 `app/shared/mineru/**` 与 `app/modules/parsing/**`：

- 提交与轮询：默认模型版本 `vlm`，开启公式与表格识别，语言 `ch`；提交后按 `mineru_poll_interval_seconds`（默认 3 秒）轮询，`mineru_poll_timeout_seconds`（默认 600 秒）超时；解析过程中通过任务心跳维持存活，避免长轮询被误判为僵尸任务。
- 解析树持久化：从 MinerU 归一化结果中提取 markdown、结构化块列表与资产文件，连同每页渲染图一并归档至 OBS，并按页、块、问题三层落库为 `parse_page` / `parse_block` / `parse_issue`，完整保留原始版面与坐标（bbox），作为后续追溯的证据层。
- 异常纠错闭环：支持页级局部重解析（`parse-versions/{id}/reparse-tasks`，仅重提指定页）与人工修正补丁（`manual-revisions`，克隆基线后合并替换），两者均生成新的 `parse_version` 并以父链指向原版本，避免整本教材重复处理。
- 解析确认：知识抽取前必须显式确认解析版本（`parse-versions/{id}/confirm`，`review_status=confirmed`），确保结构化只从「已确认可用」的版本推进。

### 3.2 知识结构化与向量化

知识结构化把碎片化版面重组为知识图谱，落点 `app/modules/knowledge/**` 与 `app/shared/vector/**`：

- 语义块与解析块二层分离：`semantic_chunk` 作为面向知识抽取的语义单元，独立于 `parse_block`（原始版面追溯层），避免直接把 MinerU 细碎块当作知识块，兼顾抽取质量与版面回溯。
- 章节边界识别 + 知识点抽取：先由 LLM 识别 markdown 章节边界、构建 `chapter_node` 章节树，再按章节分片抽取知识点（定义、例题、练习、公式），并写入 `knowledge_evidence` 记录每个知识点到原始证据的映射。抽取在语义块级并发执行（详见 3.4）。
- 向量化写入 Milvus：通过独立 Embedding 服务为知识点与语义块生成向量，写入两个逻辑集合 `semantic_chunk_vector`、`knowledge_point_vector`；集合 schema、向量维度（`milvus_embedding_dim`）、索引类型（默认 HNSW）与度量（默认 COSINE）均由配置驱动，集合落点 `app/shared/vector/client.py`，支持 schema 一致性校验与幂等建表。
- 知识人工修正：采用补丁式版本提交（`update_summary` / `update_chapter` / `add_point` / `update_point` / `delete_point` / `merge_points`），生成新知识版本并重建向量。

### 3.3 结构化 LLM 接入

所有生成环节共用一套 OpenAI 兼容结构化输出服务，落点 `app/shared/llm/service.py` 与 `app/shared/llm/client.py`：

- 双格式可切换：`LLM_API_FORMAT=response` 调用 `/responses`，使用 `text.format.type=json_schema`；`LLM_API_FORMAT=chat` 调用 `/chat/completions`，使用 `response_format.type=json_object`，以适配 DeepSeek 等不支持 Responses API 的网关。两种格式都复用「只输出 JSON」提示词，并以 Pydantic 响应模型做最终校验。
- strict schema 兼容：Responses 模式下可将 Pydantic 输出的 JSON Schema 自动清洗为 OpenAI Structured Outputs strict 兼容形态（补齐 `additionalProperties=false` 与 `required`、剥除 strict 不支持的约束关键字与 `$ref` 兄弟键），从而开启严格结构化解码。
- 三层健壮性：传输层瞬时错误重试（客户端，`llm_max_retries` 默认 2、指数退避基数 `llm_retry_base_seconds`）、缺文本重试（重发原始调用）、解析/校验失败时回调 LLM 自修复 JSON（`llm_parse_repair_max_attempts` 默认 2）。
- 推理强度与流式安全：配置 `LLM_REASONING_EFFORT` 后按格式分别注入 `reasoning.effort` 或 `reasoning_effort`，否则改用 `temperature`；SSE 流式错误 detail 按 `llm_stream_error_detail_max_chars`（默认 4096）截断，避免敏感细节外泄。
- 多模态可选：`LLM_MULTIMODAL_ENABLED` 默认关闭、零影响；开启后教案生成可注入不超过 `llm_multimodal_max_images`（默认 6）张证据图片。

### 3.4 并发与 Prompt Cache 加速

生成与抽取的吞吐由线程级并发与上游前缀缓存共同保障：

- 语义块级并发抽取：知识抽取使用 `ThreadPoolExecutor`（线程前缀 `knowledge-extract`），并发上限 `parallel_limit = min(KNOWLEDGE_EXTRACT_MAX_CONCURRENCY, 语义块数)`，默认并发 10（范围 1-10，置 1 等价串行）。落点 `app/modules/knowledge/tasks.py`。
- 教案课次级并发 + 暖缓存：教案生成先串行生成第 1 课为上游建立 prompt cache，再对剩余课次以 `ThreadPoolExecutor`（线程前缀 `lesson-plan`）并发，并发上限 `parallel_limit = min(LESSON_PLAN_MAX_CONCURRENCY, 剩余课次)`，默认并发 10。落点 `app/modules/lesson_plan/tasks.py`。
- Prompt cache 机制：落点 `app/shared/llm/prompt_cache.py`，按业务键派生 `prompt_cache_key`（前缀 `eduweave`），让多账号代理稳定命中同一前缀缓存分片；Anthropic 兼容端额外在稳定前缀末条挂 `cache_control` 标记，OpenAI 兼容端靠稳定前缀自动缓存；每次调用通过 `on_usage` 回调聚合 `cached_tokens`，便于度量命中效果。

### 3.5 生成编排

生成链由「一键生成」编排与「批次根」共同驱动：

- 一键生成状态机：`orchestrator` 模块以 `generation_run` 驱动跨阶段编排，状态为 `pending → running → waiting_user_confirm → succeeded / failed`，活跃状态为 `(pending, running, waiting_user_confirm)`；同一项目同时仅允许一个活跃 run，重复触发幂等返回当前 run。各阶段通过解析成功、解析确认、知识成功等钩子自动推进，`auto_confirm_parse` 决定解析后是否需要人工确认。落点 `app/modules/orchestrator/service.py`。
- 批次根冻结基线：`generation_batch` 冻结 `knowledge_version_id` 与 `learner_profile_version_id`、章节范围、课次与测评策略；课程大纲与教案成功后回写批次，测评、课件与覆盖率结果按同一批次查询，保证多成果物同源一致与血缘可查。落点 `app/modules/pipeline/**`、`app/modules/curriculum/**`。
- 编排顺序：批次自动顺序编排课程大纲、多课次教案与初始覆盖率分析；测评蓝图、试卷与 Raccoon 课件由用户按需触发并复用同一批次；课件成功后刷新覆盖率，覆盖率刷新成功后批次方进入 `success`。

### 3.6 课件生成（Raccoon PPT）

课件生成对接 Raccoon PPT OpenAPI，落点 `app/modules/courseware/**` 与 `app/shared/ppt/**`：

- 提交与短轮询：根据教案构建幻灯片结构后提交远程任务，按 `raccoon_poll_interval_seconds`（默认 5 秒）短轮询，`raccoon_short_poll_timeout_seconds`（默认 120 秒）未完成则停泊。
- 停泊态与自动归档：未及时返回时停泊在 `waiting_raccoon_result`，需用户补充信息时停泊在 `waiting_user_input`；Celery beat 周期复查停泊在 `waiting_raccoon_result` 的任务并自动把 PPTX 归档至 OBS，因此关闭页面也能完成；`waiting_user_input` 需用户调用回复接口后继续。用户亦可手动 `refresh` 刷新。

### 3.7 测评、覆盖率与生成追溯

- 测评生成：`assessment` 模块按场景预设（默认 `unit_test`）生成测评蓝图与试卷题目，落库 `assessment_blueprint`、`paper_result`、`question_item`，支持试卷 Word 导出；`homework` 复用共享的题目考查依据装配逻辑（`app/shared/question_basis.py`）。
- 覆盖率分析：`coverage` 模块以规则统计构建知识点覆盖矩阵，对比生成题目与教材原始知识点，输出覆盖率、难度分布与质量告警，落库 `coverage_report`。
- 生成追溯：`generation_trace` 记录成果物到知识点/教材的来源与贡献度，支撑「题目 → 知识点 → 语义块 → 解析块 → 原页位置」的全链路回溯与影响分析。

### 3.8 异步任务框架

长流程统一由任务中心治理，落点 `app/modules/task_center/**` 与 `app/worker.py`：

- 任务记录：`task_record` 记录任务生命周期（`pending → processing → success / failure`），`task_step_record` 记录细粒度步骤进度；按业务键（biz_key）保证同项目同类型任务同时仅一个活跃实例，实现幂等。
- 心跳与僵尸回收：worker 在长操作前后更新心跳；beat 的 `system.reap_stale_tasks` 按 `task_stale_threshold_seconds`（默认 1800 秒）回收卡在 `processing` 的僵尸任务，等待外部异步结果的停泊任务不会被误判。
- 错误分类与重试：可重试错误（LLM、MinerU、Raccoon 等外部依赖瞬时类）按 `retry_count`/`max_retry_count` 指数退避自动重排（基数 `task_retry_backoff_base_seconds` 默认 30 秒），业务校验类错误直接判失败。
- 课次级断点续跑：多课次教案先写入 `lesson_plan_generation_item` 中间结果，全部成功后再批量转正式 `lesson_plan`；失败任务可按课次断点续跑，已成功课次跳过（单课次业务级重试由 `lesson_plan_session_max_retries` 控制，默认 2）。

### 3.9 存储与文件

文件资产统一存放华为云 OBS，落点 `app/shared/storage/**` 与 `app/modules/file_asset/**`。源文件、解析工件、页图、导出文档均以 `file_object` 登记并按业务前缀归档；对外通过通用文件下载接口（`files/{file_object_id}/download-url`）下发签名地址，有效期由 `obs_signed_url_expire_seconds`（默认 3600 秒）控制。

## 4. 系统性能与工程化说明

### 4.1 性能目标与达成路径

系统设定的关键质量与耗时目标如下：

| 指标 | 目标 |
| --- | --- |
| 教材结构识别完整率 | ≥ 90% |
| 章节树识别准确率 | ≥ 90% |
| 公式还原可用率 | ≥ 85% |
| 表格还原可用率 | ≥ 85% |
| 知识点抽取命中率 | ≥ 85% |
| 题目知识点关联准确率 | ≥ 85% |
| 重点知识点覆盖率 | ≥ 90% |
| 难度分布偏差 | ≤ 15% |
| 教材解析任务完成时长（200 页） | ≤ 8 分钟 |
| 端到端生成时长 | ≤ 20 分钟 |
| 覆盖率报告完整性 | = 100% |

达成路径主要依靠三个工程手段：

- 异步化：所有长流程（解析、抽取、生成、分析）均异步执行并细分步骤，前端通过任务中心轮询进度，避免请求阻塞。
- 并发化：知识抽取按语义块、教案按课次并发（默认各 10 路线程），把原本串行的多次 LLM 调用压缩到单批耗时量级。
- 缓存化：教案先暖缓存再并发，复用上游前缀缓存显著降低重复前缀的 token 成本与时延。

### 4.2 可靠性工程

- 幂等：按业务键约束同类任务唯一活跃实例，重复触发不产生重复副本。
- 崩溃恢复：心跳 + reaper 周期回收僵尸任务并重排，停泊任务不被误判。
- 失败重试：外部依赖瞬时错误指数退避自动重试，业务错误快速失败并明确归因。
- 停泊复查：课件远程任务由 beat 周期复查自动完成，关闭页面不影响交付。
- 断点续跑：多课次教案按课次中间结果续跑，避免整批重做。

### 4.3 可维护性与配置体系

- 模块化小分层：每个业务模块默认包含 `router` / `service` / `repository` / `schemas` / `models` / `tasks` / `tests`，跨模块优先经由对方 service 协作，外部依赖统一走 `shared`。
- 数据库迁移：以 Alembic 为正式初始化与演进入口（`alembic upgrade head`），SQL 脚本作为历史参考与对齐入口，库表结构变更同步维护 `sql/` 下脚本。
- 环境隔离与配置：后端统一使用 `backend/.venv` 独立虚拟环境，避免 `numpy`/`pymilvus` 等二进制依赖污染全局；全部外部依赖与阈值通过环境变量集中配置（`app/core/config.py`），并对关键项做取值校验。
- 健康检查：提供 `/health` 与 `/ready` 探针，便于容器编排与就绪判定。

### 4.4 质量保障

- 自动化测试基线：当前 261 项自动化测试全部通过。
- 严格 schema 探针：`scripts/probe_*` 探测各生成环节的严格结构化 schema 可用性。
- 真实链路回归：`scripts/run_*realtest*.py` 等脚本对教案并发、生成断点续跑、Raccoon 课件等关键能力做真实环境回归（产物见第 6 章）。

## 5. 应用实现的能力

EduWeave 以单角色（教师）贯穿全链路，交付一套从原始素材到成品资源的完整工作流。系统对外提供 RESTful 接口，前端源码尚未入库，本章从能力层面描述系统已实现的功能与交互。

### 5.1 端到端能力清单

- 项目工作台：项目创建与上下文管理，串联教材、学情、知识与生成成果。
- 教材多版本与解析复核：PDF 上传与多版本共存、MinerU 解析、页级证据浏览、异常列表、局部重解析与人工修正、解析确认。
- 班级学情聚合画像：按班级上传多名学生 DOCX，本地解析个体画像后由 LLM 聚合为班级学情画像（共同难点、学习风格分布、目标分层与教学建议），支持人工修正。
- 知识图谱编辑：章节树与知识点浏览、证据回溯、补丁式人工修正与版本管理。
- 一键生成：从解析到成果物的跨阶段自动编排，支持是否自动确认解析、课次数量与课时时长等参数。
- 成果物生成与管理：课程大纲、分课次教案、PPT 课件、课后作业、单元/期末试卷、覆盖率报告，均支持查询、预览与按需触发。
- 任务中心：任务列表、步骤级进度、失败归因与课次级重试。
- 导出与下载：课程大纲、教案、试卷等支持 Word 导出，文件统一通过签名地址下载。

### 5.2 前端能力层面

前端定位为面向教师的资源生产工作台，规划采用 React 18 + TypeScript + Vite + React Router v7 + Zustand + TanStack Query v5 + shadcn/ui + Tailwind CSS 技术栈。其与后端的交互模式包括：以 TanStack Query 对接 RESTful 接口、对长流程任务做轮询展示进度、对解析与知识结构提供结构化编辑视图、对成果物提供预览与导出入口。生成过程的聚合展示由后端 `generation_process` 模块提供面向前端的聚合数据。

### 5.3 关键 API 面

系统已稳定的接口按域组织，统一以 `/api/v1` 为前缀，并提供 `/health`、`/ready` 健康探针。代表性接口如下：

- 认证：`POST /auth/login`、`GET /auth/me`
- 项目与教材学情：`/projects/**`、`/projects/{id}/textbooks`、`/projects/{id}/learner-profiles`、`/learner-profile-versions/{id}`
- 教材解析与复核：`/textbook-versions/{id}/parse-tasks`、`/parse-versions/{id}/confirm`、`/parse-versions/{id}/reparse-tasks`、`/parse-versions/{id}/manual-revisions`
- 知识结构化：`/parse-versions/{id}/knowledge-tasks`、`/knowledge-versions/{id}`、`/knowledge-versions/{id}/chapters`、`/knowledge-versions/{id}/points`、`/knowledge-points/{id}`、`/knowledge-versions/{id}/manual-revisions`
- 生成与成果物：`/generation-batches`、`/curriculum-plans`、`/lesson-plans`、`/assessment-blueprints`、`/paper-results`、`/courseware-results`（含 `/refresh`、`/reply`）、`/coverage-reports`
- 任务与文件：`/tasks/**`、`/files/{file_object_id}/download-url`

## 6. 创新与突破点（含验证说明）

本章每个创新点给出实现要点与验证支撑。验证以现有自动化测试基线、真实链路回归报告与需求规格质量指标为依据。

### 6.1 生成基线冻结，保障多成果物同源一致

`generation_batch` 在生成阶段一次性冻结知识版本、学情版本、章节范围与生成策略，使课程大纲、教案、课件、试卷、覆盖率报告全部来自同一组基线，从根本上规避传统资源生产中「各成果物版本错配」的问题。验证：批次以 `knowledge_version_id`、`learner_profile_version_id` 等血缘字段固化基线，真实链路回归中批次 `success` 且各成果物按同一批次查询一致。

### 6.2 语义块与解析块二层分离

`semantic_chunk` 面向知识抽取、`parse_block` 面向原始版面追溯，二者分离既保证知识抽取质量，又保留完整版面证据，使知识点可回溯到教材原页位置。验证：知识版本详情可逐知识点展开证据映射；两类向量集合 `semantic_chunk_vector`、`knowledge_point_vector` 分别承载块级与点级检索。

### 6.3 全链路生成追溯

`generation_trace` 配合共享的题目考查依据装配（`question_basis`），打通「题目 → 知识点 → 语义块 → 解析块 → 原页位置」的追溯链，支撑覆盖率分析、考查依据卡片渲染与影响分析。验证：覆盖率报告对生成题目与教材知识点逐项比对并输出覆盖矩阵与告警。

### 6.4 异常纠错闭环

「自动识别异常 → 页级局部重解析 → 人工修正 → 新版本生效」的闭环，让复杂版式教材的纠错聚焦到出问题的局部，避免整本重处理。验证：`reparse-tasks` 与 `manual-revisions` 均生成带父链的新解析版本，确认后方可进入结构化。

### 6.5 课次级并发 + Prompt Cache + 断点续跑

教案生成「先暖缓存、再并发、可续跑」三位一体，在保证产出一致性的同时显著提升吞吐与成本效率。验证（教案生成并发真实测试）：7 课次任务中，第 1 课串行暖缓存（约 156 秒）后，其余 6 课在 `lesson-plan_0..5` 线程并发执行；`cache_warmup_completed=True`，LLM 用量 `total_tokens=110461`、其中 `cached_tokens=55040`（前缀缓存命中约一半 token），任务状态 `success`。

### 6.6 任务自愈与停泊态远程复查

僵尸任务由 reaper 周期回收重排，课件远程任务由 beat 周期复查自动归档，二者共同保证「崩溃可恢复、关闭页面也能完成」。验证（生成 resume 真实测试）：在真实 45 页教材（知识版本含 14 章节、148 知识点）上复用解析与学情重跑知识抽取与大纲教案生成，生成批次最终 `success`，验证全链路可恢复重跑。

### 6.7 验证总览

- 自动化基线：261 项自动化测试全部通过。
- 真实链路回归：教案并发报告与生成 resume 报告（见 6.5、6.6）。
- 质量目标：需求规格第 10.7 节质量与耗时指标作为验收基线（见第 4.1 节）。

## 7. 适用场景与应用价值

### 7.1 适用场景

- 单教材备课：教师上传一本教材，快速产出整本的大纲、教案、课件与测练。
- 多教材/多版本：同一项目下教材多版本共存，按需切换基线重新生成。
- 单班/多班教学：学情以班级为单位聚合多名学生画像，生成贴合班级特征的个性化资源；不同班级可基于同一教材生成差异化成果。
- 跨学科适配：解析与结构化基于通用版面与语义能力，理科公式、表格、实验图表等复杂版式同样适用。
- 系统集成前景：以 RESTful 接口与对象存储为边界，可作为教学资源生产中台对接现有教务/备课系统。

### 7.2 应用价值

EduWeave 针对教育办公中的典型痛点给出系统性解法。这些痛点包括：教材版式复杂，传统文本抽取难以复原；解析后知识碎片化，难以转换为教学逻辑；学情分散，难以融入教学规划；从教材到成果物高度依赖人工，生产效率低；资源缺乏来源追溯，难以做质量控制。对应的价值如下：

- 降本增效：把「万字教材到精美课件/测练题库」的长链路从人工密集转为自动化流水线，显著压缩备课与组卷工时。
- 质量可控：覆盖率与难度分布分析让考点覆盖与难度配比可视化、可校验，测评有据可依。
- 个性化：融合班级学情画像，使大纲、教案与测练贴合真实学情而非千篇一律。
- 可追溯：从成果物到教材原页的证据链，使每一处生成内容都可解释、可复核。
- 可复用、可扩展：版本化与批次化的数据资产沉淀，使教学资源成为可持续复用与二次加工的数字资产。

## 8. 总结与后续演进

EduWeave 已打通「教材到课堂」的全链路生产闭环：以 MinerU 为高保真解析入口、MySQL 为业务数据中心、OBS 为文件资产中心、Milvus 为语义检索中心、大模型与 Raccoon PPT 为生成引擎，配合版本血缘、生成基线冻结、全链路追溯、并发与缓存、任务自愈等工程能力，形成可复用、可扩展、可追溯、可控制的智能教学资源生产体系，并以 261 项自动化测试与真实链路回归报告作为质量背书。

后续演进沿 P0 → P1 → P2 平滑推进：

- 生成能力：多 PPT 模板、试卷 PDF 导出、混合检索（关键词 + 向量）。
