# EduWeave 前端开发阶段上下文

## 文档说明

本文档用于记录前端各 Phase 的实现上下文、关键决策、验证结果、已知风险和后续输入，作为下一阶段开发的交接材料。

记录规则：

- 每个 Phase 结束后追加记录，不覆盖历史。
- 只记录工程上下文和接口事实，不记录真实 token、AK/SK、JWT 或本地 `.env` 内容。
- 若阶段内发现后端契约、环境依赖或演示路径变化，需要在对应 Phase 的“已知风险”和“交给下一 Phase 的输入”中写清楚。

## Phase 1：前端地基与任务诊断

### 阶段目标

- 补齐成果类 API client 和 TypeScript 类型。
- 新增任务详情页，用于真实联调时诊断异步任务失败原因。
- 新增生成批次详情基础页，作为后续成果页的入口。
- 建立可延续的阶段上下文记录机制。
- 补充仓库忽略规则，避免提交 `node_modules`、`dist` 和本地构建缓存。

### 当前输入

- 前端本地端口：`http://127.0.0.1:7777`
- 后端 API：`http://127.0.0.1:8010`
- 赛题材料目录：`/Users/seven/Downloads/教育赛题`
- 赛题材料不在 Phase 1 导入，Phase 2 再用于真实上传联调。

### 关键决策

- Phase 1 不新增 mock 数据，不改后端接口。
- 批次详情页只做基础信息、关联任务和成果入口，深度成果展示留到 Phase 4。
- 任务详情页在 Phase 1 做完整诊断视图，优先支持后续外部服务缺配置时的排错。
- JSON 字段暂按 `Record<string, unknown> | null` 展示，不在前端假设深层业务结构。
- Embedding 是否需要由后端确认；前端只展示真实任务错误，不在 Phase 1 绕过或掩盖失败。

### 已完成产物

- `.gitignore` 已补充 Node、前端构建产物和 TypeScript 构建缓存忽略规则。
- 前端 API client 已补齐课程大纲、教案、测评、课件、覆盖率、导出下载等成果接口。
- 前端类型已补齐任务步骤、任务详情、文件下载、课程大纲、教案、测评蓝图、试卷、课件和覆盖率报告。
- 新增 `/tasks/:taskId` 任务详情页，支持任务概览、步骤时间线、错误信息、载荷和结果 JSON 展示。
- 新增 `/projects/:projectId/batches/:batchId` 生成批次基础页，支持批次基线、关联任务和成果入口空状态。
- 任务表已增加“详情”入口，生成批次列表已可进入批次详情页。
- 新增任务轮询判断、进度条、错误提示和 JSON 查看器等共享能力。

### 验证结果

- `npm run build` 通过。
- `git status --short --untracked-files=all | rg "frontend/(node_modules|dist)|node_modules|dist/"` 未发现 `node_modules` 或 `dist` 待提交。
- 阶段上下文文档和前端源码未写入真实密钥。

### Review 修复记录

- 批次详情页已改为以后端返回的 `generation_batch.project_id` 作为项目查询和返回链接依据，避免手动输入错误 `projectId` 时页面指向错误项目。
- 清理了任务轮询工具中暂未使用的导出函数，避免 Phase 1 遗留无消费抽象。
- Review 修复后 `npm run build` 通过。

### 已知风险

- 后端知识抽取当前代码包含向量写入步骤；若 Embedding 配置缺失，该任务可能在 `upsert_vectors` 阶段失败。
- 外部服务 token、OBS、MinerU、LLM、Raccoon 的连通性需要在 Phase 2 真实联调中验证。

### 交给 Phase 2 的输入

- Phase 2 可以基于现有项目工作台继续重构教材区、学情区、知识区、生成区和任务区。
- Phase 2 开始使用 `/Users/seven/Downloads/教育赛题` 做真实教材与学情上传联调。
- 若知识抽取在 `upsert_vectors` 阶段失败，优先从 `/tasks/:taskId` 查看后端返回的真实错误码、错误信息和步骤明细。
- 批次详情页已有基础入口，Phase 4/5 可继续补课程方案、教案、测评、课件、覆盖率和导出下载的深度展示。

## Phase 2：知识抽取主链路与项目工作台闭环

### 阶段目标

- 将项目工作台重构为教材上传与解析、学情上传与抽取、知识抽取、生成批次四段流程。
- 每段展示当前选中版本、真实状态、最近相关任务和任务详情入口。
- 按后端真实依赖控制按钮可用性，减少无效请求和重复任务。
- 完成项目 `#1` 的真实知识抽取端测，验证知识版本可以进入 `ready` 状态。

### 当前输入

- 前端本地端口：`http://127.0.0.1:7777`
- 后端 API：`http://127.0.0.1:8010`
- 真实项目：项目 `#1`
- 教材解析版本：解析版本 `#1`，101 页，2408 blocks，已确认。
- 学情版本：学情版本 `#1`，`extract_status=success`，`version_status=ready`。

### 关键决策

- 前端不硬编码赛题文件路径，仍通过浏览器文件选择器手动上传。
- Phase 2 不新增 mock 数据，不改后端接口，不做成果详情深度展示。
- 创建批次前要求知识版本为 `ready`，学情版本同时满足 `extract_status=success` 与 `version_status=ready`。
- 知识抽取前要求解析版本已成功并已确认，避免直接触发后端会拒绝的请求。
- 已有知识版本时，重新抽取按钮传 `force_regenerate=true`；否则按首次抽取处理。
- 知识抽取任务处于 `pending`、`running`、`processing` 时禁用按钮，避免重复点击创建并发任务。

### 已完成产物

- `/projects/:projectId` 已改为四段式工作台流程。
- 教材、学情、知识、生成批次区域均显示最近相关任务、进度、错误摘要和 `/tasks/:taskId` 入口。
- 上传、解析、确认、抽取、创建批次的按钮已按真实依赖显示禁用原因。
- 任务中心分页数量扩大到 30 条，便于联调时看到近期异步任务。
- `StatusBadge` 已补充 `ready`、`uploaded`、`archived`、`unknown` 状态展示。
- 解析任务创建策略已改为后端真实支持的 `mineru_vlm_default`。
- 知识版本刷新后优先选择最新 `ready` 版本，避免停留在已归档旧版本。
- 已有知识版本时按钮文案显示为“重新抽取知识”。

### 真实端测结果

- API 级知识抽取测试通过：`tests/test_knowledge_api.py` 为 `6 passed`。
- 通过浏览器项目页点击“重新抽取知识”，创建知识抽取任务 `#10`。
- 任务 `#10` 最终状态为 `success`，`current_stage=upsert_vectors`，`progress_percent=100`。
- 任务 `#10` 无 `last_error_code`，无 `last_error_message`。
- `prepare_parse_source` 成功：`page_count=101`，`block_count=2408`。
- `invoke_llm_extract` 成功：`chapter_count=11`，`point_count=165`，`llm_call_count=19`。
- `persist_knowledge_result` 成功：生成知识版本 `#2`，11 章，165 个知识点。
- `upsert_vectors` 成功：18 个语义片段向量，165 个知识点向量。
- 最新知识版本为 `#2`，`version_status=ready`。
- 数据落库结果：11 个章节，165 个知识点，427 条证据；旧知识版本 `#1` 已归档。

### 前端验证结果

- `npm run build` 通过。
- 浏览器项目页确认知识区显示 `165 个知识点`、`就绪`。
- 知识版本列表显示 `知识 #2` 为就绪，`知识 #1` 为已归档。
- 最近知识任务显示 `#10` 成功，进度 100%。
- 生成批次基线显示知识版本 `#2` 就绪、学情版本 `#1` 就绪。
- “创建批次”按钮在基线满足时可用。
- Phase 2 前端 review 后已修复：重抽后优先选中新 ready 知识版本、运行中禁用重复抽取、按钮文案区分首次抽取与重新抽取。

### 已知风险

- 后端标准 Celery worker 入口仍需后端同学确认模型预加载问题；前端不修改后端源码。
- `.env` 中 `LLM_API_FORMAT=response` 是本地真实链路配置，不作为前端提交内容。
- 当前本地仍有未跟踪文件：`README.md`、`docker-compose.yml`、`docs/知识抽取真实联调问题记录.md`；Phase 2 本次提交不混入这些文件。

### 交给 Phase 3 的输入

- Phase 3 可直接使用知识版本 `#2` 和学情版本 `#1` 创建生成批次。
- Phase 3 重点是证明结构化知识可以继续生成课程大纲、教案、作业/测试、课件和覆盖率报告。
- 若生成任务失败，优先从工作台最近任务入口进入任务详情页查看步骤、错误码和后端原始信息。
- 批次详情页已有基础入口，后续可继续补课程方案、教案、测评、课件、覆盖率和导出下载的深度展示。
