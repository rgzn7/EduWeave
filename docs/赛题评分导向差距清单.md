# 赛题评分导向差距清单

- 编写日期：2026-05-19
- 目标：以比赛拿到好名次为目标，倒推 EduWeave 还需要补齐的产品能力、后端接口、前端展示和演示材料。
- 当前基线：Phase3-D 已完成本地演示打磨，核心页为 `http://127.0.0.1:7777/projects/1/batches/1`。

## 1. 总体判断

当前系统已经能支撑“教材 + 学情 -> 知识结构 -> 课程方案 -> 教案 -> 单元测评 -> 课件 PPTX -> 覆盖报告”的真实演示主线。

但从赛题拿分角度，下一阶段不能只满足“单元测评跑通”和“批次详情可看”。更高分的作品需要让评委明显感受到：

- 这是一个完整的“教材到课堂”生产系统，而不是几组接口拼接。
- MinerU 解析结果确实被转化为可追溯、可复用的教学资产。
- 测练不是一次性单元卷，而是可以沉淀、扩展、复用的测练体系。
- 覆盖报告不是装饰统计，而是能校验生成成果与原始知识点的质量控制闭环。

## 2. 必须补齐

### 2.1 测练体系从 unit_test 升级为多场景测练

赛题要求：自动生成匹配大纲的课后作业、期末测试卷等。

当前状态：

- 后端已有通用测评接口：`POST /api/v1/curriculum-plans/{curriculum_plan_id}/assessment-tasks`。
- 请求体支持 `assessment_strategy_json.scenario_type` 和 `assessment_strategy_json.scene_type`。
- 真实验收只跑通 `unit_test`。
- 前端只暴露“生成单元测评”。

要补的后端：

- 明确定义并固化至少 3 个测练场景预设：
  - `homework`：课后作业，题量少，贴合本课/本章节。
  - `unit_test`：单元测评，当前已跑通。
  - `final_exam` 或 `comprehensive_exam`：综合/期末测试，题量更多，覆盖面更广。
- 为每个场景配置默认 `question_count`、`question_types`、`difficulty_range` 和 prompt 约束。
- 补真实任务端测，至少证明 `homework` 和 `unit_test` 可成功生成。
- 保证同一批次下不同 `scene_type` 可以分别生成和查询，不互相冲突。

要补的前端：

- 测评 Tab 从单按钮改成场景选择：
  - 课后作业
  - 单元测评
  - 综合测试
- 列表按场景展示已有结果，允许切换查看蓝图、试卷、题目、DOCX。
- 概览页把“测评”从单个状态升级为“测练体系”摘要，例如：作业 1 份、单元卷 1 份、综合卷等待生成。

验收标准：

- 在项目 `#1` 批次 `#1` 中，至少能看到 `unit_test` 和另一个测练场景的真实结果。
- 每个结果都有题目、答案解析、知识点来源摘要和 DOCX 下载。

### 2.2 覆盖报告纳入测评题目与课件结果

赛题要求：通过对比生成题目与教材原始知识点，展示考点覆盖是否全面、难度分布是否合理。

当前状态：

- 覆盖报告已经能展示覆盖率、告警、成果物覆盖摘要。
- 现有后端说明里，覆盖分析主要覆盖课程方案和本批次教案。
- 测评题目和课件页面是否纳入覆盖统计，需要后端确认并补齐。

要补的后端：

- 覆盖分析输入扩展为：
  - 课程方案知识点引用
  - 教案知识点引用
  - 试卷题目 `question_item.knowledge_point_id`
  - 课件结构中的知识点引用或页面类型统计
- `coverage_summary_json` 或 `report_json.artifact_coverage` 中明确输出各类成果物覆盖情况。
- 输出难度分布校验，至少基于题目 `difficulty_level` 统计并与测评策略范围对比。

要补的前端：

- 覆盖报告 Tab 展示“成果物覆盖矩阵”：
  - 课程方案
  - 教案
  - 试卷题目
  - 课件
- 对测评部分展示题型分布、难度分布、未覆盖知识点。
- 概览页把覆盖校验描述从“等待覆盖分析”升级为“课程/教案/测练/课件覆盖校验”。

验收标准：

- 覆盖报告能明确说明测评题目覆盖了哪些知识点、遗漏了哪些知识点、难度分布是否合理。
- 演示时可以从覆盖报告跳回测评或任务详情，讲清楚质量控制闭环。

### 2.3 教材解析证据可视化

赛题要求：基于 MinerU 对复杂排版、公式、实验流程图等教材文档进行深度拆解。

当前状态：

- 系统已接入 MinerU，真实记录中有解析版本、页数、block 数和知识版本。
- 前端项目工作台能展示教材/解析/知识的核心状态。
- 但演示主线更偏生成成果，解析证据可视化还不够有冲击力。

要补的后端：

- 确认现有解析详情接口是否能返回：
  - 页级文本
  - block 类型统计
  - 原始页码
  - 图片/表格/公式相关 block 或资源引用
  - MinerU 参数，例如是否开启公式、表格、OCR
- 如果已有字段，给前端一份稳定响应示例。
- 如果没有，至少补一个解析版本摘要接口或在详情中补充统计字段。

要补的前端：

- 在项目工作台或解析详情中加入“解析证据”视图：
  - 页数、block 数、章节数、知识点数
  - block 类型统计
  - 知识点来源证据
  - 原始页码/片段引用
- 演示时用这块证明 MinerU 不只是被调用，而是真的支撑后续知识结构。

验收标准：

- 评委能在页面上看到“教材 PDF 被解析成结构化证据”，而不是只看到一个成功状态。

## 3. 强烈建议补齐

### 3.1 题库基础视图

赛题关键词：测练题库。

当前状态：

- 后端已有 `question_item` 落库。
- 前端在测评详情页展示题目，但还没有独立题库视图。

要补的后端：

- 如果现有接口足够，确认 `GET /paper-results/{id}` 中的 `questions` 是否可以作为题库基础说明。
- 更理想的是新增查询接口：
  - 按项目/批次查题目
  - 按知识点查题目
  - 按题型/难度筛选

要补的前端：

- 轻量版本：在测评 Tab 增加“题目沉淀”区域，按知识点、题型、难度展示题目。
- 强版本：新增项目级“题库”视图，展示本项目下生成过的题目。

验收标准：

- 至少能讲清楚：生成题目不是一次性文本，而是结构化落库，可按知识点、难度、题型复用。

### 3.2 课程方案和教案 DOCX 下载入口

当前状态：

- 后端已有课程方案和教案 DOCX 导出接口封装。
- 前端批次详情成果 Tab 重点展示内容，但当前演示重点验证的是试卷 DOCX 和课件 PPTX。

要补的前端：

- 在课程方案 Tab 增加“导出 DOCX”入口。
- 在教案 Tab 增加“导出 DOCX”入口。
- 无 `export_file_id` 时可触发后端导出；已有文件时直接获取签名下载地址。

验收标准：

- 课程方案、教案、试卷、课件四类成果均可下载或导出，形成完整交付感。

### 3.3 比赛演示主叙事页或脚本

当前状态：

- `/projects/1/batches/1` 已能作为演示主页面。
- 但页面仍是工程批次视角，比赛讲解需要更强叙事。

要补的前端或材料：

- 轻量方式：写固定演示脚本，明确每一步点击哪里、讲什么。
- 强方式：新增一个只读“演示总览”入口，把链路整理成：
  - 输入基线
  - MinerU 解析证据
  - 知识结构
  - 学情适配
  - 教学资源包
  - 测练体系
  - 覆盖校验
  - 成果下载

验收标准：

- 非技术评委不需要理解任务 ID，也能在 3-5 分钟内看懂系统价值。

## 4. 可以包装说明，不建议当前硬做

### 4.1 完整 audit 审计视图

原因：

- 价值高，但不是比赛核心评分点。
- 需要后端审计事件模型、埋点和查询接口，成本较大。

建议口径：

- 当前已有任务详情、步骤明细、覆盖报告和轻量 `generation_trace`。
- 完整审计视图作为后续产品化能力。

### 4.2 Raccoon 补充问题回复真实验收

原因：

- 前端和接口已预留。
- 真实触发依赖 Raccoon 远端行为，不稳定，不适合当前冲刺。

建议口径：

- 保留为远端任务交互能力，不作为本次主线验收。

### 4.3 学生端做题与自动判分

原因：

- 赛题重点是教师端资源生产，不是学生端学习闭环。
- 会明显拉大范围。

建议口径：

- 本期聚焦教师备课与资源生成，学生端练习闭环作为后续扩展。

## 5. 后端对接清单

请后端同学优先确认以下问题：

1. `assessment_strategy_json.scene_type` 除 `unit_test` 外，是否可以稳定支持 `homework` 和 `final_exam`？
2. 是否愿意在后端固化测练场景预设，避免前端随意拼策略导致生成质量不可控？
3. 覆盖报告当前是否统计 `question_item` 和 `courseware_result.structure_json`？如果没有，预计怎么补？
4. 是否能提供解析版本摘要：页数、block 数、block 类型统计、图片/表格/公式统计、MinerU 参数？
5. `question_item` 是否可以作为题库基础能力对外说明？是否需要新增题目查询接口？
6. 课程方案 DOCX、教案 DOCX、试卷 DOCX、课件 PPTX 四类下载接口是否都可作为比赛可承诺能力？
7. 当前演示数据中，是否有更适合展示复杂教材解析的 PDF 和解析结果？

## 6. 前端冲刺清单

若后端接口确认可用，前端建议按这个顺序补：

1. 测评 Tab 增加场景选择和多测练结果展示。
2. 课程方案和教案 Tab 补 DOCX 导出入口。
3. 覆盖报告 Tab 增强成果物覆盖矩阵展示。
4. 项目工作台补 MinerU 解析证据视图。
5. 增加题目沉淀/题库基础展示。
6. 写比赛演示脚本和提交材料说明。

## 7. 优先级建议

P0，直接影响赛题符合度：

- 多场景测练生成。
- 覆盖报告纳入测评题目。
- 解析证据可视化。
- 演示脚本。

P1，明显提高作品完整度：

- 题库基础视图。
- 课程方案/教案 DOCX 导出入口。
- 覆盖报告纳入课件结构。

P2，当前不抢：

- 完整 audit 审计视图。
- Raccoon 补充问题真实触发验收。
- 学生端做题与判分。

## 8. 推荐下一步

下一步不是直接开 PR，而是先和后端用本清单对齐 P0：

1. 后端确认测练多场景和覆盖报告扩展的接口可行性。
2. 前端同步设计测评 Tab 和覆盖报告 Tab 的最小增强方案。
3. 选一条真实演示数据重新端测，确保不是只在文档上成立。
