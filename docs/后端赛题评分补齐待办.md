# 后端赛题评分补齐待办

- 编写目的：明确后端为了支撑比赛展示还需要补齐的能力。
- 当前判断：核心生成链路已基本完成，剩余工作重点是让前端更容易展示、让评委更容易理解、让演示数据更稳定。
- 范围说明：本清单只覆盖后端待办，不包含前端页面实现。

## 1. 解析证据摘要接口

### 目标

为前端提供稳定的 MinerU 解析证据摘要，证明教材 PDF 已被拆解为可追溯、可复用的结构化资产。

### 建议接口

```http
GET /api/v1/parse-versions/{parse_version_id}/evidence-summary
```

### 建议返回内容

- 解析版本基础信息：
  - `parse_version_id`
  - `textbook_version_id`
  - `strategy_code`
  - `mineru_model`
  - `parse_status`
  - `review_status`
  - `version_status`
- 规模统计：
  - `page_count`
  - `block_count`
  - `issue_count`
  - `asset_block_count`
- block 类型统计：
  - `heading`
  - `paragraph`
  - `table`
  - `image`
  - `equation`
  - `formula`
  - 其他实际出现的 block 类型
- 多媒体与结构统计：
  - 图片 block 数
  - 表格 block 数
  - 公式/方程 block 数
  - 带 `asset_file_id` 的 block 数
  - 带坐标 `bbox_json` 的 block 数
- MinerU 参数摘要：
  - 是否启用 OCR
  - 是否启用公式解析
  - 是否启用表格解析
  - 对应 `strategy_code` 解析出的策略参数
- 示例证据：
  - 示例页码
  - 示例 block 编号
  - block 类型
  - 文本片段
  - 坐标
  - 资源文件 ID

### 验收标准

- 前端不需要分页拉取全部解析页，也能展示页数、block 数、block 类型分布和 MinerU 参数。
- 至少能返回 3 到 10 条示例 evidence block，便于页面展示“原文片段 -> 结构化块”的证据链。
- 当前教师只能访问自己项目下的解析版本。

## 2. 题库查询接口

### 目标

基于已落库的 `question_item` 提供轻量题库查询能力，支撑前端做“题目沉淀/题库基础视图”。

### 建议接口

```http
GET /api/v1/question-items
```

### 建议筛选参数

- `generation_batch_id`
- `paper_result_id`
- `knowledge_point_id`
- `question_type`
- `difficulty_level`
- `scene_type`
- `page`
- `page_size`

### 建议返回字段

- 题目基础信息：
  - `id`
  - `generation_batch_id`
  - `paper_result_id`
  - `knowledge_point_id`
  - `question_no`
  - `question_type`
  - `difficulty_level`
  - `score_value`
- 题目内容：
  - `stem_text`
  - `options_json`
  - `answer_text`
  - `analysis_text`
  - `source_trace_json`
- 所属试卷摘要：
  - `paper_title`
  - `scene_type`
  - `assessment_blueprint_id`
- 时间字段：
  - `created_at`
  - `updated_at`

### 验收标准

- 可按批次查询本批次全部题目。
- 可按知识点查询相关题目。
- 可按题型、难度和测练场景筛选。
- 不越权返回其他教师项目下的题目。

## 3. 演示数据校准

### 目标

保证比赛演示环境中已有足够样例，不需要现场临时生成所有内容。

### 建议确认项

- 项目 `#1` / 批次 `#1` 至少包含：
  - `unit_test`
  - `homework` 或 `final_exam`
- 覆盖报告已刷新，并包含以下四类成果物：
  - 课程方案
  - 教案
  - 试卷题目
  - 课件页面
- 解析版本能展示多种 block 类型，最好包含：
  - 标题
  - 段落
  - 图片
  - 表格
  - 公式或方程
- 四类成果物导出链路可用：
  - 课程方案 DOCX
  - 教案 DOCX
  - 试卷 DOCX
  - 课件 PPTX

### 验收标准

- 演示前执行一次固定检查脚本或手工检查清单，确认上述数据存在。
- 覆盖报告中的 `artifact_coverage` 能看到 `question_item` 和 `courseware_slide`。
- DOCX/PPTX 下载地址能正常打开。

## 4. 文档同步

### 目标

避免 README、API 文档或对接文档继续保留旧口径，影响前端开发和比赛材料描述。

### 需要同步的描述

- 测评不再只是默认 `unit_test`，后端已支持多场景测练。
- 测练场景由后端预设固化，前端只传 `scene_type`。
- 试卷 DOCX 已支持导出。
- 覆盖报告已纳入：
  - `question_item`
  - 课件页面知识点引用
- `question_item` 可作为题库基础能力说明。
- 课程方案、教案、试卷 DOCX 和课件 PPTX 均可作为比赛交付能力说明。

## 5. 不需要重复投入的后端能力

以下能力当前不建议重复开发，只需要保持回归测试通过：

- 多场景测练预设。
- 覆盖报告统计试卷题目和课件页面。
- 课程方案 DOCX 导出接口。
- 教案 DOCX 导出接口。
- 试卷 DOCX 导出接口。
- 课件 PPTX 下载。
- 完整审计视图。
- 学生端答题和自动判分。

- 按 `knowledge_point_id` 查询题目。
- 按 `question_type` 查询题目。
- 按 `difficulty_level` 查询题目。
- 按 `scene_type` 查询题目。
