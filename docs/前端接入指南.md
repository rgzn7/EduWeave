## 一、后端本地启动

### 1.1 前置依赖

| 依赖 | 版本要求 | 说明 |
|------|---------|------|
| Python | ≥ 3.12 | 建议使用 pyenv 管理 |
| MySQL | 8.0 | 数据库 |
| Redis | 任意稳定版 | 任务队列与缓存 |
| Milvus | 2.x | 向量数据库，可用 Docker 启动 |

### 1.2 环境变量配置

进入 `backend/` 目录，复制示例文件并按实际填写：

```bash
cp .env.example .env
```

最少需要填写的字段：

```env
# 数据库（本地 MySQL 8.0，默认 root/boss1114）
MYSQL_HOST=127.0.0.1
MYSQL_USER=root
MYSQL_PASSWORD=boss1114
MYSQL_DATABASE=eduweave

# Redis
REDIS_URL=redis://127.0.0.1:6379/0

# JWT（本地开发随意填）
JWT_SECRET=local_dev_secret

# LLM（需要对接实际模型才能触发生成）
LLM_API_BASE_URL=https://api.openai.com/v1
LLM_API_KEY=your_key
LLM_MODEL=your_model_name
```

### 1.3 启动步骤

```bash
cd backend/

# 创建虚拟环境并安装依赖
python -m venv .venv
source .venv/bin/activate
pip install -e ".[dev]"

# 执行数据库迁移
python -m alembic upgrade head

# 初始化演示账号和 Milvus 集合
python scripts/bootstrap_local.py

# 启动开发服务（默认监听 0.0.0.0:8000）
python scripts/start_dev.py
```

### 1.4 接口文档地址

| 地址 | 说明 |
|------|------|
| `http://localhost:8000/docs` | Swagger UI，可直接在线调试所有接口 |
| `http://localhost:8000/redoc` | ReDoc 风格文档，适合阅读 |
| `http://localhost:8000/health` | 存活检查 |
| `http://localhost:8000/ready` | 依赖就绪检查（MySQL/Redis/Milvus） |

### 1.5 CORS 说明

后端默认允许 `http://localhost:5173` 和 `http://127.0.0.1:5173`（Vite 默认端口）跨域访问。若前端使用其他端口，需在 `.env` 中追加：

```env
CORS_ALLOWED_ORIGINS=http://localhost:5173,http://localhost:3000
```

## 二、认证流程

### 2.1 演示账号

| 字段 | 值 |
|------|----|
| 用户名 | `teacher_demo` |
| 密码 | `Teacher@123` |

### 2.2 登录获取 Token

```http
POST /api/v1/auth/login
Content-Type: application/json

{
  "username": "teacher_demo",
  "password": "Teacher@123"
}
```

响应示例：

```json
{
  "success": true,
  "code": 200,
  "message": "操作成功",
  "data": {
    "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "token_type": "Bearer",
    "expires_in": 7200,
    "user": {
      "id": 1,
      "username": "teacher_demo",
      "display_name": "示例教师",
      "role_code": "teacher",
      "status": "active"
    }
  },
  "timestamp": "2026-05-10T10:00:00.000000Z",
  "request_id": "a5c0f6b6-1234-5678-9abc-def012345678"
}
```

### 2.3 携带 Token 发起请求

所有需要认证的接口，在 HTTP 请求头中添加：

```http
Authorization: Bearer {access_token}
```

### 2.4 Token 过期处理

- 默认有效期 120 分钟
- 过期后接口返回 `401` + 错误码 `TOKEN_EXPIRED`
- 前端收到 `TOKEN_EXPIRED` 后，跳转到登录页重新登录

### 2.5 获取当前用户信息

```http
GET /api/v1/auth/me
Authorization: Bearer {access_token}
```

## 三、统一请求响应格式

### 3.1 成功响应

```json
{
  "success": true,
  "code": 200,
  "message": "操作成功",
  "data": { },
  "timestamp": "2026-05-10T10:00:00.000000Z",
  "request_id": "a5c0f6b6-1234-5678-9abc-def012345678"
}
```

创建类接口（POST）成功时 HTTP 状态码为 `201`，code 字段仍为 `201`。

### 3.2 分页列表响应

```json
{
  "success": true,
  "code": 200,
  "message": "获取列表成功",
  "data": {
    "items": [ ],
    "pagination": {
      "total_count": 125,
      "page": 1,
      "page_size": 20,
      "total_pages": 7,
      "has_previous": false,
      "has_next": true
    }
  },
  "timestamp": "2026-05-10T10:00:00.000000Z",
  "request_id": "a5c0f6b6-1234-5678-9abc-def012345678"
}
```

分页查询参数：`?page=1&page_size=20`（page_size 最大 100）

### 3.3 错误响应

```json
{
  "success": false,
  "code": 401,
  "message": "用户名或密码错误",
  "data": null,
  "timestamp": "2026-05-10T10:00:00.000000Z",
  "request_id": "a5c0f6b6-1234-5678-9abc-def012345678",
  "errors": [
    {
      "code": "INVALID_CREDENTIALS",
      "message": "用户名或密码错误",
      "details": null,
      "field": null
    }
  ]
}
```

参数验证失败（422）时，`errors` 数组中每项的 `field` 字段指向出错的参数名。

## 四、错误码清单

前端应根据 `errors[0].code` 进行精确的错误分支处理。

### 认证类（HTTP 401/403）

| 错误码 | HTTP | 说明 |
|--------|------|------|
| `UNAUTHORIZED` | 401 | 未携带 Token |
| `INVALID_TOKEN` | 401 | Token 格式无效 |
| `TOKEN_EXPIRED` | 401 | Token 已过期，需重新登录 |
| `INVALID_CREDENTIALS` | 401 | 用户名或密码错误 |
| `ACCOUNT_DISABLED` | 403 | 账号已禁用 |
| `RESOURCE_FORBIDDEN` | 403 | 无权访问此资源 |

### 资源不存在类（HTTP 404）

| 错误码 | 说明 |
|--------|------|
| `PROJECT_NOT_FOUND` | 项目不存在 |
| `TEXTBOOK_NOT_FOUND` | 教材版本不存在 |
| `LEARNER_PROFILE_NOT_FOUND` | 学情文件不存在 |
| `PARSE_VERSION_NOT_FOUND` | 解析版本不存在 |
| `TASK_NOT_FOUND` | 任务不存在 |
| `FILE_NOT_FOUND` | 文件不存在 |
| `KNOWLEDGE_VERSION_NOT_FOUND` | 知识版本不存在 |
| `GENERATION_BATCH_NOT_FOUND` | 生成批次不存在 |
| `CURRICULUM_PLAN_NOT_FOUND` | 课程大纲不存在 |
| `LESSON_PLAN_NOT_FOUND` | 教案不存在 |
| `ASSESSMENT_BLUEPRINT_NOT_FOUND` | 测评蓝图不存在 |
| `PAPER_RESULT_NOT_FOUND` | 试卷结果不存在 |
| `HOMEWORK_BLUEPRINT_NOT_FOUND` | 课后作业蓝图不存在 |
| `HOMEWORK_RESULT_NOT_FOUND` | 课后作业结果不存在 |
| `HOMEWORK_QUESTION_NOT_FOUND` | 课后作业题目不存在 |
| `COURSEWARE_RESULT_NOT_FOUND` | 课件结果不存在 |
| `COVERAGE_REPORT_NOT_FOUND` | 覆盖率报告不存在 |

### 业务约束类（HTTP 409/422）

| 错误码 | HTTP | 说明 |
|--------|------|------|
| `TASK_CONFLICT` | 409 | 同类型任务已在运行，不允许重复触发 |
| `PARSE_VERSION_NOT_CONFIRMED` | 422 | 解析版本未确认，无法进行知识抽取 |
| `GENERATION_BASELINE_INVALID` | 422 | 生成批次的知识版本或学情版本无效 |
| `PROJECT_REFERENCE_INVALID` | 422 | 项目引用的版本主键不合法 |
| `INVALID_FILE_TYPE` | 422 | 文件类型不支持 |
| `INVALID_PARSE_STRATEGY` | 422 | 解析策略不合法 |
| `INVALID_PAGE_RANGE` | 422 | 页码范围格式不合法 |
| `KNOWLEDGE_REVISION_INVALID` | 422 | 知识修正操作参数不合法 |

### 外部服务类（HTTP 503/504）

这类错误通常是后台任务失败时返回，前端显示"服务暂时不可用，请稍后重试"即可。

| 错误码 | HTTP | 说明 |
|--------|------|------|
| `EXTERNAL_SERVICE_ERROR` | 503 | 外部服务异常 |
| `LLM_REQUEST_FAILED` | 503 | LLM 调用失败 |
| `MINERU_SUBMIT_FAILED` | 503 | 教材解析服务提交失败 |
| `MINERU_POLL_TIMEOUT` | 504 | 教材解析服务轮询超时 |
| `RACCOON_REQUEST_FAILED` | 503 | 课件生成服务调用失败 |
| `RACCOON_POLL_TIMEOUT` | 504 | 课件生成服务轮询超时 |
| `FILE_UPLOAD_FAILED` | 503 | 文件上传到对象存储失败 |
| `DEPENDENCY_NOT_READY` | 503 | 后端依赖服务（数据库/Redis 等）未就绪 |

## 五、业务主流程

### 5.1 整体流程图

```
[创建项目]
    ↓
[上传教材 PDF]  →  [创建解析任务]  →  [轮询任务直到 success]  →  [确认解析版本]
                                                                        ↓
[上传学情 DOCX] →  [轮询学情抽取任务直到 success]
                                                                        ↓
                        [创建知识抽取任务]  →  [轮询任务直到 success]
                                                                        ↓
                        [创建生成批次]  →  自动生成课程大纲 & 教案
                                       ↓
                        （可选）[触发测评生成]  →  查看试卷结果
                        （可选）[触发课件生成]  →  查看课件结果
                                                                        ↓
                                              [导出 DOCX] / [查看覆盖率]
```

---

### 5.2 步骤详解

#### 步骤 1：创建项目

```http
POST /api/v1/projects
Authorization: Bearer {token}
Content-Type: application/json

{
  "name": "小学数学秋季提升班",
  "subject_code": "math",
  "grade_code": "grade_3"
}
```

保存响应中的 `data.id` 作为 `project_id`。

---

#### 步骤 2：上传教材

```http
POST /api/v1/projects/{project_id}/textbooks
Authorization: Bearer {token}
Content-Type: multipart/form-data

file: <PDF文件>
set_as_current: true
```

保存响应中的 `data.id` 作为 `textbook_version_id`。

---

#### 步骤 3：创建教材解析任务

```http
POST /api/v1/textbook-versions/{textbook_version_id}/parse-tasks
Authorization: Bearer {token}
Content-Type: application/json

{
  "strategy_code": "vlm",
  "set_as_current_on_success": true
}
```

保存响应中的 `data.id` 作为 `task_id`，然后进行任务轮询（见第七节）。

---

#### 步骤 4：查看解析版本并确认

任务 `task_status` 为 `success` 后，查询解析版本列表：

```http
GET /api/v1/textbook-versions/{textbook_version_id}/parse-versions
Authorization: Bearer {token}
```

取最新版本的 `id` 作为 `parse_version_id`，然后确认：

```http
POST /api/v1/parse-versions/{parse_version_id}/confirm
Authorization: Bearer {token}
```

确认后该解析版本才能用于知识抽取。

---

#### 步骤 5：上传学情

```http
POST /api/v1/projects/{project_id}/learner-profiles
Authorization: Bearer {token}
Content-Type: multipart/form-data

file: <docx文件>
auto_extract: true
set_as_current: true
```

`auto_extract: true` 时会自动发起抽取任务，响应中包含 `task_id`，同样进行轮询。

保存响应中学情版本的 `id` 作为 `learner_profile_version_id`（从版本详情接口获取）。

---

#### 步骤 6：创建知识抽取任务

```http
POST /api/v1/parse-versions/{parse_version_id}/knowledge-tasks
Authorization: Bearer {token}
Content-Type: application/json

{
  "force_regenerate": false
}
```

轮询任务，成功后查询知识版本：

```http
GET /api/v1/parse-versions/{parse_version_id}/knowledge-versions
Authorization: Bearer {token}
```

保存最新 `knowledge_version_id`。

---

#### 步骤 7：创建生成批次（核心编排入口）

```http
POST /api/v1/generation-batches
Authorization: Bearer {token}
Content-Type: application/json

{
  "project_id": 1,
  "knowledge_version_id": 1,
  "learner_profile_version_id": 1,
  "batch_name": "第一轮课程规划",
  "course_count": 12,
  "session_duration_minutes": 90
}
```

此接口会自动触发课程大纲和教案的生成任务。保存响应中的 `data.id` 作为 `generation_batch_id`。

批次状态通过 `GET /api/v1/generation-batches/{generation_batch_id}` 查询，其中 `tasks` 数组包含所有关联任务的状态。

---

#### 步骤 8：查看课程大纲与教案

生成完成后：

```http
# 课程大纲列表
GET /api/v1/curriculum-plans?project_id={project_id}

# 教案列表（需 curriculum_plan_id）
GET /api/v1/lesson-plans?curriculum_plan_id={curriculum_plan_id}
```

---

#### 步骤 9：触发测评生成（可选）

测评生成按场景拆为两条入口：单元测试 / 期末综合测走批次级，课后作业按课次级生成。

```http
# 单元测试或期末综合测（scene_type 仅支持 unit_test / final_exam）
POST /api/v1/curriculum-plans/{curriculum_plan_id}/assessment-tasks
Authorization: Bearer {token}
Content-Type: application/json

{
  "scene_type": "unit_test"
}
```

生成完成后：

```http
# 测评蓝图
GET /api/v1/assessment-blueprints?curriculum_plan_id={curriculum_plan_id}

# 试卷结果
GET /api/v1/paper-results?generation_batch_id={generation_batch_id}
```

---

#### 步骤 9.5：触发课后作业生成（可选，每课一份）

课后作业按教案维度生成，每份教案最多一份成功作业；与单元测试 / 期末综合测互不影响，可并存。课后作业当前不在自动 pipeline 中批量生成，前端应在教案列表或教案详情页按 `lesson_plan_id` 单独触发。

```http
POST /api/v1/lesson-plans/{lesson_plan_id}/homework-tasks
Authorization: Bearer {token}
```

响应为任务中心的 `TaskListItem`。本地同步任务模式下，`data.result_json` 可能立即包含 `homework_result_id`、`homework_blueprint_id`、`question_count`、`lesson_plan_id`；异步模式下，前端应根据 `data.id` 走任务中心轮询，直到 `task_status=success` 后再查询作业详情。

生成完成后：

```http
# 按教案获取本课唯一作业（包含题目明细）
GET /api/v1/lesson-plans/{lesson_plan_id}/homework-result

# 大纲下所有课次作业一览（前端"作业总览"用，按 class_session_no 升序）
GET /api/v1/homework-results?curriculum_plan_id={curriculum_plan_id}

# 单份作业详情
GET /api/v1/homework-results/{homework_result_id}

# 作业题目筛选（题库化场景）
GET /api/v1/homework-questions?lesson_plan_id={lesson_plan_id}
```

前端页面建议：

- 教案列表页：按每条教案展示“生成作业 / 查看作业”操作；已存在成功作业时不要重复触发生成。
- 教案详情页：优先调用 `GET /lesson-plans/{lesson_plan_id}/homework-result`，若返回 `HOMEWORK_RESULT_NOT_FOUND` 再展示生成入口。
- 作业总览页：使用 `GET /homework-results?curriculum_plan_id={curriculum_plan_id}` 聚合全部课次作业，列表字段中的 `class_session_no`、`lesson_title` 可作为主要排序和展示依据。
- 题库筛选页：使用 `GET /homework-questions`，按 `knowledge_point_id`、`question_type`、`difficulty_level` 做筛选。

常见错误处理：

- `409 TASK_CONFLICT`：该课已有成功作业或已有运行中作业生成任务，前端刷新作业详情或任务状态即可。
- `404 HOMEWORK_RESULT_NOT_FOUND`：该课尚未生成作业，前端展示生成按钮。
- `404 LESSON_PLAN_NOT_FOUND`：当前账号无权访问该教案，或教案不存在。
- `422 ASSESSMENT_SCENE_INVALID`：批次级 `assessment-tasks` 接口不再接受 `scene_type=homework`，课后作业必须改走 `/lesson-plans/{lesson_plan_id}/homework-tasks`。

---

#### 步骤 10：触发课件生成（可选）

```http
POST /api/v1/lesson-plans/{lesson_plan_id}/courseware-tasks
Authorization: Bearer {token}
```

课件生成依赖外部 Raccoon PPT 服务，状态可能需要多次刷新：

```http
POST /api/v1/courseware-results/{courseware_result_id}/refresh
Authorization: Bearer {token}
```

---

#### 步骤 11：导出 DOCX

```http
# 导出课程大纲
POST /api/v1/curriculum-plans/{curriculum_plan_id}/export-docx

# 导出教案
POST /api/v1/lesson-plans/{lesson_plan_id}/export-docx

# 导出试卷
POST /api/v1/paper-results/{paper_result_id}/export-docx

# 导出课后作业
POST /api/v1/homework-results/{homework_result_id}/export-docx
```

响应中包含签名下载 URL，直接用于文件下载。

## 六、关键 ID 依赖关系

| 子资源 | 所需上游 ID | 来源接口 |
|--------|------------|---------|
| `textbook_version_id` | `project_id` | `POST /api/v1/projects/{project_id}/textbooks` |
| 解析任务 `task_id` | `textbook_version_id` | `POST /api/v1/textbook-versions/{id}/parse-tasks` |

| 学情文件 | `project_id` | `POST /api/v1/projects/{project_id}/learner-profiles` |
| `learner_profile_version_id` | `profile_file_id` | `GET /api/v1/projects/{pid}/learner-profiles/{fid}/versions` |
| 知识任务 `task_id` | `parse_version_id`（已确认）| `POST /api/v1/parse-versions/{id}/knowledge-tasks` |
| `knowledge_version_id` | `parse_version_id` | `GET /api/v1/parse-versions/{id}/knowledge-versions` |
| `generation_batch_id` | `project_id` + `knowledge_version_id` + `learner_profile_version_id` | `POST /api/v1/generation-batches` |
| `curriculum_plan_id` | `project_id`（批次完成后）| `GET /api/v1/curriculum-plans?project_id=` |
| `lesson_plan_id` | `curriculum_plan_id` | `GET /api/v1/lesson-plans?curriculum_plan_id=` |
| `assessment_blueprint_id` | `curriculum_plan_id` | `GET /api/v1/assessment-blueprints?curriculum_plan_id=` |
| `paper_result_id` | `generation_batch_id` | `GET /api/v1/paper-results?generation_batch_id=` |
| `homework_result_id` | `lesson_plan_id` 或 `curriculum_plan_id` | `GET /api/v1/lesson-plans/{lesson_plan_id}/homework-result` / `GET /api/v1/homework-results?curriculum_plan_id=` |
| `homework_blueprint_id` | `homework_result_id` | 详情接口响应中包含；或通过 `GET /api/v1/homework-blueprints/{id}` 直接获取 |
| `courseware_result_id` | `generation_batch_id` | `GET /api/v1/courseware-results?generation_batch_id=` |
| `coverage_report_id` | `generation_batch_id` | `GET /api/v1/coverage-reports?generation_batch_id=` |
| 文件下载 URL | `file_object_id` | `GET /api/v1/files/{id}/download-url` |

## 七、任务轮询说明

### 7.1 轮询接口

```http
GET /api/v1/tasks/{task_id}
Authorization: Bearer {token}
```

### 7.2 task_status 枚举

| 值 | 说明 |
|----|------|
| `pending` | 等待执行 |
| `running` | 执行中 |
| `success` | 执行成功 |
| `failed` | 执行失败 |

### 7.3 轮询策略建议

- 每 **3-5 秒**轮询一次
- 超过 **200 次**（约 10-15 分钟）未结束可提示用户超时
- 状态变为 `success` 或 `failed` 后停止轮询
- `failed` 时可从 `last_error_code` 和 `last_error_message` 获取失败原因

### 7.4 任务列表查询

```http
GET /api/v1/tasks?project_id={project_id}&module_code=parsing
Authorization: Bearer {token}
```

`module_code` 可选值：`parsing`、`knowledge`、`curriculum`、`lesson_plan`、`assessment`、`homework`、`courseware`、`coverage`、`learner_profile`

### 7.5 生成批次任务总览

```http
GET /api/v1/generation-batches/{generation_batch_id}
Authorization: Bearer {token}
```

响应中的 `tasks` 数组包含该批次下所有任务的状态，适合做批次进度汇总展示。

## 八、文件上下行说明

### 8.1 文件上传

教材和学情文件使用 `multipart/form-data` 格式上传：

```http
POST /api/v1/projects/{project_id}/textbooks
Content-Type: multipart/form-data

file: <PDF>           # 必填，教材 PDF
set_as_current: true  # 可选，上传后设为当前教材
```

```http
POST /api/v1/projects/{project_id}/learner-profiles
Content-Type: multipart/form-data

file: <DOCX>          # 必填，支持 .doc/.docx
auto_extract: true    # 可选，自动发起学情抽取
set_as_current: true  # 可选，上传后设为当前学情
```

### 8.2 文件下载

所有文件通过签名 URL 下载，**不需要携带 Authorization 头**：

```http
GET /api/v1/files/{file_object_id}/download-url
Authorization: Bearer {token}
```

响应：

```json
{
  "data": {
    "download_url": "https://obs.cn-east-3.myhuaweicloud.com/...",
    "expires_in": 3600
  }
}
```

签名 URL 有效期 3600 秒，前端直接用此 URL 发起下载或展示即可。

### 8.3 DOCX 导出下载

导出接口直接返回签名 URL，无需单独查文件：

```http
POST /api/v1/curriculum-plans/{id}/export-docx
Authorization: Bearer {token}
```

响应中的 `data.download_url` 即为可直接下载的地址。
