# EduWeave 数据库详细设计说明书

- 文档版本：v1.0
- 编写日期：2026-05-17
- 作者：xisy / Codex
- 文档状态：可落地版本
- 适用范围：EduWeave P0 后端 MySQL 结构设计与 Milvus 向量库设计

## 1. 文档说明

本文档用于承接现有的需求规格、后端概要设计和后端模块详细设计，输出可直接落地的数据库方案。本文档的目标不是继续讨论“是否需要某张表”，而是明确：

- P0 阶段 MySQL 需要落哪些表。
- 每张表承担什么职责，和哪个模块对应。
- 哪些数据放 MySQL，哪些放 OBS，哪些放 Milvus。
- 版本链、批次链、任务链和追溯链如何落库。

本次设计以“主链路可跑通、结构可扩展、避免过度范式化”为原则，最终确定：

- MySQL：`28` 张表
- Milvus：`4` 类 collection，其中 `2` 类为 P0 必做，`2` 类为 P1 预留

## 2. 设计原则

### 2.1 总体原则

- 关系型数据库只保存“业务真相”和可筛选、可关联、可审计的数据骨架。
- 复杂层级内容优先使用 JSON 字段保存，避免 P0 过度拆表。
- 原始文件、中间文件和导出文件统一放 OBS，数据库只保存引用。
- 向量统一放 Milvus，不在 MySQL 中保存 embedding 向量。
- 所有版本化对象都保留历史版本，不做覆盖式更新。
- 生成链路统一用 `generation_batch` 冻结输入基线，避免下游模块版本错配。
- 结果子表默认通过直接父对象确定项目归属，不重复保存可稳定推导的 `project_id` 或跨层外键；仅在高频查询确有必要时保留少量冗余字段。

### 2.2 本次取舍

以下对象本次明确不单独建表：

- `textbook_document`
  原因：P0 阶段教材先按项目内版本管理，不额外拆教材主档。
- `teaching_objective`
  原因：教学目标摘要先放入 `knowledge_version.summary_json` 或 `curriculum_plan.content_json`。
- `parse_revision_record`
  原因：P0 阶段先用 `parse_version.parent_parse_version_id`、`diff_json` 和 `parse_issue` 表达重解析与人工修订链。

### 3.2 表数量说明

本次之所以定为 `28` 张表，是因为采用了如下策略：

- 主链路对象全部关系化。
- 新增 `semantic_chunk` 作为知识抽取后的语义块主表，避免直接把 MinerU 的细碎原始块当作教学知识块。
- 复杂内容优先放 JSON，避免再拆十几张明细表。
- 题目级、任务步骤级、片段追溯级三类"增强对象"一并纳入 P0，保证这版设计是完整可扩展的，不需要后面推倒重来。

> 增量说明：2026-05-25 拆分课后作业到课次维度，新增 `homework_blueprint`、`homework_result`、`homework_question` 共 3 张表，当前数据库总表数为 `31` 张。`paper_result` / `assessment_blueprint` / `question_item` 仅服务 `unit_test` 与 `final_exam`；课后作业全链路落入 homework_* 三张表，唯一键收紧到 `lesson_plan_id` 维度，详见 4.6 / 4.7。

### 4.4 教材解析

#### 4.4.1 `parse_version`

作用：

- 管理全量解析、局部重解析、人工确认后的解析版本链。
- 作为知识抽取唯一输入基线。

关键字段说明：

- `parent_parse_version_id`：支持页级重解析或修订版本回溯。
- `page_range_text`：记录局部重解析页范围。
- `strategy_code`、`mineru_model`：记录解析策略和模型配置。
- `diff_json`：保存相较父版本的差异摘要。

#### 4.4.2 `parse_page`

作用：

- 保存页级文本、页级状态、页级布局摘要。
- 支撑页级预览、页级重解析和页码定位。

#### 4.4.3 `parse_block`

作用：

- 保存块级结构结果，如标题、正文、公式、表格、图片。
- 作为 MinerU 原始解析结果保留页内位置、坐标和图片资源引用。
- 不再作为知识抽取后的主语义块；知识抽取产生的教学语义块落到 `semantic_chunk`。

#### 4.4.4 `parse_issue`

作用：

- 管理缺页、乱码、公式截断、表格断裂等异常。
- 记录人工处理状态和关联重解析结果。

### 4.5 知识结构

#### 4.5.1 `knowledge_version`

作用：

- 管理知识结构版本。
- 绑定唯一 `parse_version`，保证知识抽取来源清晰。

#### 4.5.2 `chapter_node`

作用：

- 保存教材一级章节节点，支持树形展示、章节范围选择、课时规划。
- `line_start`、`line_end` 记录章节在页级拼接 Markdown 中的行号范围，章节前的封面、目录等内容不落章节节点。

#### 4.5.3 `semantic_chunk`

作用：

- 保存后端基于 LLM 章节起始行号切出的教材语义块。
- 作为教材片段向量化和知识点证据映射的主对象。
- 通过 `source_block_refs_json` 保留来源 `parse_block`、页码、块号、坐标和资源文件引用，保证语义块仍能回到教材原位置。

说明：

- `semantic_chunk.chunk_text` 是后续写入 `semantic_chunk_vector` 的主要文本来源。
- `semantic_chunk.line_start`、`semantic_chunk.line_end` 记录语义块自身的行号范围；常规情况下继承章节范围，章节内容过长时按 UTF-8 字节上限继续拆成多个片段，以避免向量库 `content` 字段超长并提升召回粒度。
- Milvus `semantic_chunk_vector.content` 不做截断；若历史语义块仍超长，写入前会展开为多条向量记录，并通过同一个 `semantic_chunk_id` 与 `segment_index`、`segment_count` 元数据保持关联。
- `parse_block` 不删除，只降级为原始版面追溯层；图片仍通过 `file_object` 与来源引用保存，不写入 Milvus。

#### 4.5.4 `knowledge_point`

作用：

- 保存知识点主体。
- 供课程大纲、试卷、覆盖率分析直接引用。

说明：

- 教学目标摘要、重点、难点、标签等先放在 `knowledge_version.summary_json`、`knowledge_point.tags_json` 与 `knowledge_point.summary_text` 中。
- 需要进一步细分时，再在 P1 拆出独立目标表。

#### 4.5.5 `knowledge_evidence`

作用：

- 记录知识点与语义块、解析页、原文片段的证据映射。
- 需要精确回到页内位置时，可继续通过 `parse_block_id` 或 `semantic_chunk.source_block_refs_json` 找到原始解析块。
- 是后续 `generation_trace` 的上游基础数据。

### 4.6 课程规划与测评

#### 4.6.1 `curriculum_plan`

作用：

- 保存课程大纲版本。
- 明确绑定 `knowledge_version` 与 `learner_profile_version`。

#### 4.6.2 `lesson_plan`

作用：

- 保存教案版本。
- 直接依赖 `curriculum_plan`，并通过 `generation_batch_id` 记录所属生成批次。
- 同一生成批次内通过 `class_session_no` 区分多份课次教案，约束同一批次课次号不重复。

#### 4.6.3 `assessment_blueprint`

作用：

- 保存单元测试与期末综合测的蓝图版本，唯一键 `(curriculum_plan_id, scenario_type, version_no)`。
- 只直接依赖 `curriculum_plan`，避免版本错配。
- 课后作业蓝图已迁出至 `homework_blueprint`，本表 `scenario_type` 仅取值 `unit_test` / `final_exam`。

#### 4.6.4 `homework_blueprint`

作用：

- 保存课后作业蓝图版本，按课次粒度独立维护，唯一键 `(lesson_plan_id, version_no)`。
- 直接绑定 `lesson_plan_id`，同时保留 `generation_batch_id` 字段供溯源与覆盖率统计使用。
- 与 `assessment_blueprint` 结构对齐（`strategy_json` / `content_json` / `version_status` 等），仅唯一键与归属维度不同。

### 4.7 生成批次与成果物

#### 4.7.1 `generation_batch`

作用：

- 作为整个生成链路的统一编排根。
- 冻结知识版本、学情版本、章节范围、模板快照、测评策略和编排选项等输入基线。
- 教材版本与解析版本通过 `knowledge_version` 链路回溯，不在批次表中重复持久化。
- 汇聚课程大纲、首份教案和覆盖率报告的结果引用；批次下全部教案通过 `lesson_plan.generation_batch_id` 反查，测评蓝图和试卷通过各自列表查询。

这是本次数据库设计中最关键的表之一。

#### 4.7.2 `courseware_result`

作用：

- 保存课件结构 JSON、模板快照和导出文件引用。
- 同一批次同一教案只保留一份课件结果，支持一个批次下为多份教案分别生成课件。

#### 4.7.3 `paper_result`

作用：

- 保存单元测试与期末综合测的试卷结构 JSON、统计摘要和导出文件引用。
- 唯一键 `(generation_batch_id, scene_type)`，`scene_type` 仅取值 `unit_test` / `final_exam`。

#### 4.7.4 `homework_result`

作用：

- 保存课后作业的结构 JSON、统计摘要和导出文件引用。
- 唯一键 `lesson_plan_id`：每份教案最多一份成功作业；同时保留 `generation_batch_id` / `homework_blueprint_id` 引用。
- 字段命名上把试卷的 `paper_json` 改名为 `content_json` 与教案 / 蓝图保持一致。

#### 4.7.5 `coverage_report`

作用：

- 保存覆盖率统计结果、告警信息、明细 JSON 和导出文件引用。
- 报告的 `artifact_coverage` 桶在 2026-05-25 起新增 `homework_question` 类型，独立统计课后作业题目对知识点的覆盖。

#### 4.7.6 `question_item`

作用：

- 保存试卷题目级结果、题型、难度、知识点标签和题目级来源依据。
- 这张表虽然可被 JSON 替代，但为了题目级筛选、题型统计、知识点关联和后续题库化，本次一并纳入。
- 仅承接 `paper_result` 维度题目（unit_test / final_exam），作业题目落入 `homework_question`。

#### 4.7.7 `homework_question`

作用：

- 保存课后作业题目明细。
- 同时持有 `homework_result_id` / `lesson_plan_id` / `generation_batch_id` 三个引用，便于按课次维度做覆盖率统计与按批次维度做题库筛选。
- 唯一键 `(homework_result_id, question_no)`。

### 4.8 任务与审计

#### 4.8.1 `task_record`

作用：

- 统一记录解析、抽取、生成、分析任务。
- 是任务中心列表和详情页的主表。

#### 4.8.2 `task_step_record`

作用：

- 记录任务步骤级状态。
- 用于展示“当前阶段”“阶段失败原因”“步骤进度条”。

#### 4.8.3 `generation_trace`

作用：

- 记录某个教案片段、课件页、题目、覆盖率条目对应的来源证据。
- 支持从结果回溯到知识点、解析块、学情记录、向量召回结果。

#### 4.8.4 `audit_log`

作用：

- 记录上传、解析、人工修正、重试、生成、导出等动作。
- 区分于 `generation_trace` 的“内容来源”，它关注“谁在什么时间做了什么操作”。

### 6.2 版本链规则

- `textbook_version` 允许多版本。
- `learner_profile_version` 允许多版本。
- `parse_version` 通过 `parent_parse_version_id` 形成重解析链。
- `knowledge_version` 通过 `parent_knowledge_version_id` 形成人工修正链。
- `curriculum_plan`、`lesson_plan`、`assessment_blueprint`、`homework_blueprint` 均按版本保留历史。
- `generation_batch` 是不可变快照，不参与默认引用切换。
- P0 阶段不为解析版本、知识版本、规划版本额外维护统一 `current_xxx_version_id`，当前使用版本通过父引用链和确认状态确定。

### 6.3 默认引用规则

`project` 在 P0 只保留三类默认引用：

- `current_textbook_version_id`
- `current_learner_profile_version_id`
- `latest_generation_batch_id`

解析版本、知识版本和规划版本不再额外引入统一默认指针；工作台回显通过最近确认版本或下游绑定对象推导即可。

### 6.4 状态值字典

当前实现中，常用状态字段采用字符串保存，并由服务层约束状态流转。主要状态值如下：

- 账号状态：`active`、`disabled`
- 项目状态：`active`、`archived`、`disabled`，其中归档和停用入口当前尚未开放
- 任务状态：`pending`、`processing`、`success`、`partial_success`、`failure`、`cancelled`
- 解析状态：`pending`、`processing`、`success`、`failure`
- 复核状态：`pending`、`confirmed`
- 版本状态：`ready`、`archived`
- 课件远程等待阶段：`waiting_raccoon_result`、`waiting_user_input`，这两类阶段属于合法停泊状态，不参与僵尸任务回收

## 8. Milvus 设计

Milvus 不属于 MySQL 表设计，但本项目已明确纳入主链路，因此必须同步定稿。

### 8.1 P0 必做 collection

#### 8.1.1 `semantic_chunk_vector`

用途：

- 存 LLM 或规则整理后的教材语义块向量。
- 供教案、课件、试题生成时召回教材证据。

建议字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 主键，格式为 `semantic_chunk:{id}`；超长语义块拆分后为 `semantic_chunk:{id}:segment:{index}` |
| `semantic_chunk_id` | 语义块主键 |
| `project_id` | 项目过滤 |
| `textbook_version_id` | 教材版本过滤 |
| `parse_version_id` | 解析版本过滤 |
| `knowledge_version_id` | 知识版本过滤 |
| `chapter_node_id` | 章节过滤 |
| `page_start` | 起始页过滤 |
| `page_end` | 结束页过滤 |
| `chunk_type` | 语义块类型 |
| `embedding_model` | embedding 模型标识 |
| `content` | 语义块正文摘要或截断文本 |
| `metadata` | 来源块、章节和追溯元数据 |
| `embedding` | 向量字段 |

#### 8.1.2 `knowledge_point_vector`

用途：

- 存知识点名称、说明、教学建议、标签摘要的向量。
- 供课程规划、教案生成、试卷生成时召回知识点证据。

建议字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 主键，格式为 `knowledge_point:{id}` |
| `project_id` | 项目过滤 |
| `knowledge_version_id` | 知识版本过滤 |
| `chapter_node_id` | 章节过滤 |
| `importance_level` | 重要度 |
| `difficulty_level` | 难度 |
| `embedding_model` | embedding 模型标识 |
| `content` | 知识点名称、说明、教学建议和标签摘要拼接文本 |
| `metadata` | 章节、路径、知识点编码和标签元数据 |
| `embedding` | 向量字段 |

当前工程基线说明：

- 运行时自动初始化只覆盖 `semantic_chunk_vector` 与 `knowledge_point_vector`
- 这两类 collection 属于 P0 必需集合，必须在本地联调与演示环境中可用

### 8.2 P1 预留 collection

#### 8.2.1 `question_sample_vector`

- 用于题型风格召回、相似题召回、题目样本增强。

#### 8.2.2 `template_fragment_vector`

- 用于课件模板片段、表达模板、页面组织样式片段的语义召回。

当前工程基线说明：

- P1 预留 collection 仅保留命名和设计约束，不在 bootstrap 阶段自动创建
- 后续进入题库增强或模板中心阶段时，再按新迁移或初始化流程补齐

### 8.3 MySQL 与 Milvus 的边界

- MySQL 保存业务对象主键、版本关系、任务状态、产物摘要、追溯结果。
- Milvus 保存向量和少量检索过滤元数据。
- 图片文件与页图不写入 Milvus，仍保存在 OBS 并通过 `file_object`、`parse_page`、`parse_block`、`semantic_chunk.source_block_refs_json` 回溯。
- 向量写入和删除由 `shared.vector` 统一完成。
- 检索结果应回写到 `generation_trace` 或结果 JSON 中，便于审计和解释。

## 9. 实施顺序建议

建议按以下顺序落地：

1. 先执行 `sys_user`、`project`、`file_object`
2. 再执行教材、学情、解析、知识结构相关表
3. 再执行课程规划、批次和成果物相关表
4. 最后执行任务、追溯、审计相关表
5. 同时初始化 Milvus 的 `semantic_chunk_vector` 与 `knowledge_point_vector`

当前基础收口阶段补充约束：

- 新环境数据库正式初始化入口为 `alembic upgrade head`
- 已由 SQL 脚本初始化、但没有 `alembic_version` 的旧库，正式对齐入口为 `python backend/scripts/reconcile_alembic.py`
- `sql/20260430_eduweave_mysql_28_tables.sql` 继续作为最终初始化脚本与历史初始化参考，不再推荐作为日常新环境初始化主入口；由于早期迁移会读取该 SQL 文件，后续修改时必须同步评估迁移回放影响，必要时应冻结迁移专用 SQL

## 10. 本次结论

本次数据库设计最终结论如下：

- MySQL 使用 `28` 张表，足以支撑 EduWeave 的 P0 主链路与后续扩展。
- `semantic_chunk` 是知识抽取后教材语义块的主表，`parse_block` 继续作为 MinerU 原始解析与版面追溯层。
- 不再额外拆分 `textbook_document`、`teaching_objective`、`parse_revision_record`。
- `generation_batch` 是生成编排根，必须独立建表并作为批次快照中心。
- `question_item`、`task_step_record`、`generation_trace` 虽然会增加表数，但能显著降低后续返工成本，建议本次一次性纳入。
- Milvus 需要同步设计，P0 至少落 `semantic_chunk_vector` 和 `knowledge_point_vector` 两类 collection。
- 当前工程已完成基础收口，Alembic 已可从空库建立完整 P0 schema，旧本地库可通过 reconcile/stamp 方式接入版本链。
