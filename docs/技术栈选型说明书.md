# 教育赛题技术栈选型说明书

- 文档版本：v0.5
- 编写日期：2026-04-11
- 作者：xisy / Codex
- 文档状态：初稿
- 适用项目：EduWeave 基于 MinerU 的“教材到课堂”全链路 AI 教学资源重构系统

## 1. 文档目的

本文档用于明确 EduWeave 项目的推荐技术栈、选型原因、实施边界与阶段性落地方案，作为后续系统架构设计、项目脚手架初始化、数据库设计、接口设计与部署方案制定的依据。

本文档基于现有需求规格说明书与赛题要求进行编写，重点解决以下问题：

- 选择什么语言和框架最适合教材解析、知识抽取、教学规划和资源生成链路。
- 如何在保证比赛交付效率的前提下，为后续扩展保留空间。
- 在前端采用 React 技术栈并兼顾比赛交付效率、界面质量和后续可维护性的前提下，如何确定合适的工程方案。

## 2. 选型原则

本项目技术选型遵循以下原则：

- 优先服务赛题主链路，即“上传 -> 解析 -> 结构化 -> 规划 -> 生成 -> 校验 -> 交付”，避免为非核心能力引入过重架构。
- 优先选择与 MinerU、文档处理、LLM 调用、PPT/Word 生成生态兼容度高的技术。
- 优先支持异步任务、长流程编排、失败重试、过程追溯和人工复核。
- 优先保证 P0 可落地，并将 Milvus 接入纳入当前主链路实现，再为多模板、多策略扩展预留空间。
- 前端优先采用成熟的可定制组件基座与轻量样式体系，在保证开发效率的同时保留足够的界面表现力和可维护性。

## 3. 推荐技术栈总览

| --- | --- | --- |

| 自动化测试 | pytest + vitest + Playwright | 后端测试、前端单测与关键流程冒烟测试 |

## 4. 后端技术栈建议

### 4.1 推荐组合

后端推荐采用 `Python 3.12 + FastAPI + Pydantic v2 + SQLAlchemy 2 + Alembic`。

### 4.2 选择原因

- MinerU、文档解析、LLM 调用、PPT/Word 生成等核心能力都更贴近 Python 生态，能够减少多语言混编带来的维护成本。
- FastAPI 适合构建结构清晰的 API 服务，接口定义、数据校验和文档生成效率高，便于快速推进比赛版本。
- Pydantic v2 非常适合处理“文档解析结果”“知识点矩阵”“课程大纲”“覆盖率报告”等结构化对象，可直接用于约束模型输出格式。
- SQLAlchemy 2 + Alembic 能兼顾工程规范和 MySQL 兼容性，适合后续进行表设计、查询优化和版本迁移。
- Python 3.12 在性能、类型支持和语法体验上较新，适合作为当前项目统一后端版本基线。

### 4.3 推荐后端依赖

建议后端在 P0 阶段优先采用以下依赖：

- `fastapi`：API 服务框架。
- `uvicorn`：开发和部署运行入口。
- `pydantic`、`pydantic-settings`：请求响应模型与配置管理。
- `sqlalchemy`、`alembic`、`pymysql`：数据库访问与迁移。
- `PyJWT` 或等价 JWT 库、`pwdlib` 或等价密码哈希库：登录认证、访问令牌签发与密码校验。
- `celery`、`redis`：异步任务编排。
- `httpx`：外部服务调用。
- `obs`：接入华为云 OBS。
- `python-docx`：课程大纲、教案、试卷等 DOCX 成果物渲染。
- `pypdf`、`PyMuPDF`：教材 PDF 读取、页图预览与局部重解析辅助。
- `pymilvus`：Milvus 向量集合初始化、写入、删除和检索。
- `structlog`：结构化日志。

### 4.4 不建议的后端方案

现阶段不建议把主后端放在 Java 或 Node.js 上，原因如下：

- 教材解析、模型编排、文档生成最终仍会回到 Python 生态，主服务若采用其他语言，容易演变为双后端并行维护。
- 当前赛题重点不是高并发交易型接口，而是长任务、可追溯、可复核和多阶段数据加工，Python 更适合快速打通主链路。

## 5. 前端技术栈建议

### 5.1 推荐组合

前端推荐采用 `React 18 + TypeScript + Vite + React Router v7 + Zustand + TanStack Query v5`，并以 `shadcn/ui + Tailwind CSS` 作为基础界面方案。

### 5.2 样式与组件策略

前端视觉和交互层建议采用“`shadcn/ui` 组件基座 + `Tailwind CSS` 样式体系 + 业务组件封装”的方案，具体如下：

- 使用 `shadcn/ui` 提供表单、弹层、表格、导航等通用组件基础能力。
- 使用 `Tailwind CSS` 负责页面和业务组件的主要样式表达，必要时配合少量自定义样式文件。
- 使用 `Radix UI` 承接底层交互和无障碍能力，减少基础交互细节的重复实现。
- 通用组件放在共享目录中，业务模块按功能拆分页面、组件、接口与类型，避免堆成单一 `components` 大目录。
- 图标建议采用 `lucide-react` 或 `Iconify`，保证风格统一和按需引入。
- 对比赛展示页保留局部深度定制能力，但不建议从零自建整套基础组件体系。

### 5.3 选择原因

- React 18 生态成熟稳定，适合当前教师工作台、任务流程、结果预览和可视化页面的快速建设。
- Vite 构建与热更新速度快，适合高频迭代的比赛开发节奏。
- Zustand 轻量且心智负担低，适合维护当前用户、当前任务、筛选条件等少量全局状态。
- TanStack Query 很适合管理“任务列表、任务状态、解析结果、生成结果、轮询更新”等服务端状态，能减少手写请求缓存和轮询逻辑。
- shadcn/ui 基于 Radix UI，交互能力成熟，同时组件代码可直接落库，便于按需调整风格和结构。
- Tailwind CSS 能显著提升页面搭建效率，尤其适合比赛周期内的快速迭代与统一视觉控制。

### 5.4 推荐前端依赖

- `react`、`react-dom`、`typescript`、`vite`：前端基础框架。
- `react-router`：页面路由管理。
- `zustand`：轻量全局状态管理。
- `@tanstack/react-query`：服务端状态和轮询。
- `tailwindcss`、`@tailwindcss/typography`：样式体系与富文本排版增强。
- `@radix-ui/*`：`shadcn/ui` 的底层交互依赖。
- `echarts`：图表展示。
- `echarts-for-react`：React 下的图表封装。
- `react-pdf`：PDF 页面预览与定位。
- `axios` 或原生 `fetch` 封装：API 调用。
- `lucide-react`：图标体系。
- `zod`：前端表单与配置校验，可选。

### 5.5 不建议的前端方案

现阶段不建议采用以下方式：

- 不建议在 P0 阶段同时混用多套大型 UI 体系，例如 `shadcn/ui` 与 Ant Design、MUI 等并行使用，否则视觉和交互规范容易失控。
- 不建议为了追求完全自主可控而从零自建整套基础组件库，当前重点应放在业务链路打通与页面质量。
- 不建议在 P0 阶段引入过多可视化拖拽搭建库，避免把精力分散到低优先级能力上。

## 6. 异步任务与流程编排

### 6.1 推荐方案

异步任务层推荐采用 `Celery + Redis`。

### 6.2 适用场景

以下任务均适合放入异步队列：

- 教材解析任务
- 学情抽取任务
- 知识点抽取任务
- 课程大纲生成任务
- 教案生成任务
- 课件生成任务
- 作业和试卷生成任务
- 覆盖率分析任务
- 重新解析与人工修正后的增量重算任务

### 6.3 选择原因

- 这些流程大多耗时较长，用户无需阻塞等待。
- Celery 在 Python 生态成熟，支持重试、失败回调、分队列处理和任务状态管理。
- Redis 能承担消息队列、短期缓存和任务状态辅助存储的角色。
- 对比赛版本而言，该方案实现成本和稳定性平衡较好。

### 6.4 P0 与 P1 边界

- P0 阶段建议前端通过轮询方式获取任务状态，降低实时推送实现复杂度。
- P1 阶段可视需要增加 SSE 推送，用于任务进度实时更新。

## 7. 数据存储与文件体系

### 7.1 MySQL 8.0

关系型主库继续采用 `MySQL 8.0`，与需求规格说明书保持一致。

适合存储的数据包括：

- 项目、教材、学情、任务、知识点、课程规划、教案、课件结构、题目蓝图、覆盖率报告、追溯记录等核心业务数据。

### 7.2 华为云 OBS

对象存储推荐继续采用 `华为云 OBS`，统一管理以下文件：

- 原始教材 PDF
- 原始学情文档
- MinerU 中间结果
- 原页截图、表格切片、公式资源
- 导出的 PPTX、DOCX；PDF 导出为后续预留能力
- 课件模板与生成快照

### 7.3 Milvus

向量检索采用 `Milvus`，并纳入 P0 主链路。

原因如下：

- 教案、课件和试题生成需要稳定的知识召回与证据引用能力，Milvus 能提供统一的向量存储与检索基础设施。
- 当前项目已明确接入 Milvus，业务模块与任务编排需要围绕向量写入、删除、检索和健康检查设计接口。
- 若后续采用 Zilliz 托管集群，仍按 Milvus 兼容协议接入，不改变业务层抽象。

## 8. AI 编排与资源生成方案

### 8.1 模型调用建议

模型调用层建议采用“OpenAI 兼容 HTTP 客户端 + 自定义服务封装 + 任务内 Prompt 组装”的方式，不建议在 P0 阶段引入过重的 Agent 编排框架。

建议做法如下：

- 用统一的模型服务层封装不同任务的调用入口。
- Prompt 当前由各任务按领域上下文组装，并由统一 LLM 服务负责结构化请求、SSE 读取、JSON 修复和 Pydantic 校验；若后续模板数量继续增加，再引入文件化模板管理。
- 使用 `Pydantic` 约束课程大纲、教案、题目、覆盖率报告等结构化输出。
- 对每次生成任务记录输入基线、模型参数、提示词版本和引用证据。

### 8.2 课件与文档生成建议

P0 阶段建议采用以下策略：

- PPTX 当前通过 Raccoon PPT OpenAPI 远程生成，后端保存结构化幻灯片 JSON、Raccoon 任务状态和归档后的 PPTX 文件引用。
- 课程大纲、教案和试卷导出当前通过 `python-docx` 直接渲染并归档到 OBS。
- PDF 相关依赖主要服务教材解析、页图预览和局部重解析；成果物 PDF 导出属于后续能力。
- 若后续需要完全本地化 PPT 模板控制，再评估 `python-pptx` 或独立 PPT 生成器方案。

### 8.3 不建议的方案

- 不建议 P0 阶段就引入复杂的多智能体编排框架，否则任务链路、日志追踪和失败定位都会变重。
- 不建议直接让模型输出最终富文档二进制结果，建议始终保留结构化中间层，便于重试、校验和追溯。

## 9. 部署与环境建议

### 9.1 开发环境

本地开发不强制使用 Docker，建议采用“本机服务 + 常规启动”的方式进行联调和开发。

建议方式如下：

- 前端使用 `npm run dev` 或等价命令启动开发服务。
- 后端 API 使用 Python 虚拟环境启动 `FastAPI` 服务。
- 异步任务使用独立终端启动 `Celery Worker`。
- MySQL 8.0 使用本机已安装服务。
- Redis 使用本机已安装服务。
- 文件存储开发阶段可直接连接测试环境 `OBS`，如需本地替代再按需接入 `MinIO`。

这种方式更适合当前项目的日常开发、调试接口、查看任务日志与快速修改配置。

### 9.2 比赛演示环境

比赛演示环境或交付环境建议采用 `Docker Compose` 或单机容器化部署，原因如下：

- 部署简单，联调成本低。
- 便于快速打包演示环境。
- 对当前项目体量而言，不必过早引入 Kubernetes。

### 9.3 可观测性建议

P0 阶段建议至少落实以下能力：

- 后端接口日志
- 异步任务日志
- 外部服务调用日志
- 任务耗时统计
- 失败原因记录
- 解析异常与人工修正记录

## 10. 推荐项目结构

项目结构建议采用“按业务模块划分，模块内再局部分层”的方式，而不是把 `api`、`models`、`services`、`repositories` 全部平铺在同一级目录下。

组织原则如下：

- 先按业务域拆模块，例如认证、项目、教材、学情、解析、知识点、课程大纲、教案、课件、测练、覆盖率、任务中心。
- 每个模块内部再放该模块自己的 `router`、`service`、`repository`、`schema`、`model`、`tests` 等文件。
- 真正跨模块复用的能力下沉到 `shared` 或 `core`，例如配置、数据库连接、存储、MinerU 封装、LLM 封装、队列、日志、中间件。
- 前后端模块命名尽量保持一致，降低接口联调和问题排查成本。
- Worker 作为独立进程启动，但业务代码仍复用后端模块，不额外复制一套目录。

## 11. 最终建议

综合当前赛题要求、现有需求文档和实施成本，推荐采用以下最终主栈：

- 前端：`React 18 + TypeScript + Vite + React Router v7 + Zustand + TanStack Query v5 + shadcn/ui + Tailwind CSS + ECharts + react-pdf`
- 后端：`Python 3.12 + FastAPI + Pydantic v2 + SQLAlchemy 2 + Alembic`
- 认证：`JWT Access Token + 密码哈希库`
- 异步任务：`Celery + Redis`
- 数据与存储：`MySQL 8.0 + 华为云 OBS`
- AI 与解析：`MinerU + OpenAI Responses json_schema / Chat Completions json_object + 任务内 Prompt 组装`
- 文档生成：`Raccoon PPT OpenAPI + python-docx`
- 向量检索：`Milvus`（如采用托管形态，可兼容 Zilliz）

这套方案的核心优势是：

- 能最快打通比赛主链路。
- 与 MinerU 和 Python 文档生态兼容度最高。
- 能较好支持异步任务、人工复核和结果追溯。
- 前端组件基础成熟，页面实现效率和展示质量更容易兼顾。
- 后续扩展到混合检索、多模板和更强生成能力时改造成本较低。
