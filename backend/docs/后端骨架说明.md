# 后端骨架说明

当前阶段已完成以下基础能力：

- FastAPI 应用启动入口
- 教师登录与当前用户接口
- `/health` 与 `/ready` 健康检查
- MySQL、Redis、Milvus 基础接入
- OBS 存储适配器
- OBS 临时签名下载地址生成能力
- Celery 队列骨架
- MinerU 官方批量解析接口适配层
- 28 表 SQL 对应的 SQLAlchemy metadata 骨架
- Alembic 完整 P0 schema 迁移链
- 已有本地库的 Alembic reconcile / stamp 对齐脚本
- 本地演示教师账号 bootstrap 脚本
- Milvus P0 必需 collection 基线收口
- 教材全量解析、页级重解析、解析人工修正版本
- 学情真实抽取、学情版本列表、学情人工修正版本
- 解析版本显式确认接口
- 知识抽取任务、知识版本列表与详情
- 章节树、知识点列表与知识点详情
- 知识人工修正版本与知识点归并
- OpenAI Responses `json_schema` 与 Chat Completions `json_object` 结构化 LLM 适配层
- 结构化 LLM 通过 `LLM_API_FORMAT` 在 `response` 与 `chat` 两种格式间切换，独立 Embedding 适配层保持不变
- OpenAI Responses 可选 `reasoning.effort` 配置注入；`chat` 格式不传入推理强度参数
- 通用文件下载地址接口
- 生成批次创建、列表与详情接口
- 生成批次自动投递课程大纲与教案生成任务
- 课程大纲 LLM 真实生成、落库与详情查询
- 教案 LLM 真实生成、落库与详情查询
- 测评蓝图 LLM 真实生成、落库与详情查询
- 单元测试试卷与题目明细落库、列表与详情查询
- Raccoon PPT 课件生成、远程状态刷新、补充问题回复与 PPTX 归档
- 覆盖率规则分析、报告落库、详情查询与轻量生成追溯写入

当前基础收口约束如下：

- 业务接口继续保持统一响应壳、全局异常处理和 JWT 依赖注入风格
- 新环境统一通过 `alembic upgrade head` 初始化数据库
- 已由 SQL 脚本初始化的旧库统一通过 `python scripts/reconcile_alembic.py` 接入 Alembic 版本链
- 本地联调统一通过 `python scripts/bootstrap_local.py` 初始化演示教师账号
- Milvus 只自动初始化 `semantic_chunk_vector`、`knowledge_point_vector` 两类 P0 必需集合
- MySQL 已新增 `semantic_chunk` 教材语义块表；Zilliz 云端集合已切换为 `semantic_chunk_vector` 与 `knowledge_point_vector`
- 知识抽取链路先用页级 Markdown 行号识别一级章节，再由后端切出 `semantic_chunk` 并逐章抽取知识点
- 阶段一默认通过 Worker 轮询 MinerU 结果，不使用回调模式
- 解析与学情人工修正采用版本级提交，不做块级 patch
- 知识抽取只允许从 `parse_status=success` 且 `review_status=confirmed` 的解析版本发起
- 知识人工修正采用操作补丁方式生成新版本，不直接覆盖历史知识版本
- 生成批次当前冻结显式传入的知识版本、学情版本、章节范围、课次和单次课时
- 课程大纲生成由 `pipeline` 统一创建批次后自动触发，当前不暴露独立生成入口，也暂不提供人工编辑版本接口
- 教案生成由课程大纲任务成功后自动触发，按 `lesson_sessions` 为每个课次生成一份教案，当前不暴露独立生成入口，也暂不提供人工编辑版本接口
- 覆盖率分析由教案任务成功后自动触发，当前只按课程大纲和本批次全部教案中的知识点引用做规则统计，不额外调用 LLM
- 测评生成改为用户按需触发，通过课程大纲创建 `assessment_generate` 任务，默认生成 `unit_test` 测评蓝图与单元测试试卷，暂不提供人工编辑版本接口和 docx/pdf 导出
- 课件生成改为用户按需触发，通过指定教案创建 `courseware_generate` 任务，当前通过 Raccoon PPT OpenAPI 创建远程任务并短轮询；若未完成则保留 `processing`，由刷新接口继续查询并归档 PPTX
- `generation_batch` 当前自动编排到 `curriculum_plan`、多份 `lesson_plan` 与 `coverage_report`，按需测评、试卷、课件仍通过同一批次根保留血缘，后续完整审计继续复用同一批次根

当前已完成第三阶段生成链路拆分，`pipeline + curriculum + lesson_plan + coverage` 为自动链，`assessment + courseware` 为按需链，并已在覆盖率报告阶段写入轻量 `generation_trace`。后续建议优先继续推进完整 `audit`，逐步把上传、解析、人工修正、生成、导出与重试动作纳入统一审计视图。
